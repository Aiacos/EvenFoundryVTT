<threat_model>
T-03-09 [HIGH]: **prom-client memory exhaustion via unbounded label cardinality** — using `session_id`, `actor_id`, or `bearer_token` as a metric label would cause `prom-client`'s internal `Map<labelHash, value>` to grow without bound. Mitigation: enforce label allowlist in `createMetricsRegistry`. Labels permitted: `method` (GET/POST + 3 others), `route` (use Fastify `request.routeOptions.url` pattern — `/v1/tools/:name` is ONE label value across all 7 tools, not 7), `status_code` (200/400/401/422/429/503), `tool` (the 7 MVP tool names). All other dimensions go in pino logs, never in metric labels. Reviewer test: `grep -E "(session_id|actor_id|bearer)" packages/bridge/src/metrics/registry.ts` MUST return 0 hits (no leak through label names).
T-03-10 [HIGH]: **prom-client global-Registry test collision** (Pitfall 2) — `new Counter({ name, help })` without `registers:[…]` registers against the process-global default Registry. Parallel test files importing `server.ts` re-execute module-top metric definitions → `Error: A metric with the name evf_http_request_duration_seconds has already been registered`. Mitigation: every metric defined inside `createMetricsRegistry(registry?)` factory which creates a fresh `new Registry()` if none supplied; every metric constructor receives `registers: [registry]`. NO module-level metric construction. Test isolation verified by `server.test.ts` running two `buildServer()` in parallel describe blocks.
T-03-11 [MEDIUM]: **/metrics endpoint leak in production** — `/metrics` returns the full Prometheus exposition with no auth (standard scrape pattern). On a public-facing bridge this leaks internal counts (active sessions, dedup rate). Mitigation: documented in JSDoc that in production, Docker Compose network policy (Plan 03-05) MUST restrict the bridge container port 8910 to the internal scrape network; the public-facing nginx/Caddy MUST NOT proxy `/metrics`. Phase 03 single-tenant homelab is LAN-only per Specs.md §11.5.3, so this is acceptable for MVP. mTLS for `/metrics` is deferred to Phase 13 (deferred per CONTEXT.md).
T-03-12 [LOW]: **/readyz reveals env-var presence** — returning `reason:'EVF_INTERNAL_SECRET_missing'` confirms to an unauthenticated scanner that the bridge expects this env var. Mitigation: this is a public infrastructure convention (k8s readiness probes are unauthenticated by design). The env-var NAME is not a secret; only its VALUE is. The `reason` string contains no value bytes. Acceptable.
</threat_model>
