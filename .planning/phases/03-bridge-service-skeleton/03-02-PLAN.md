<threat_model>
T-03-05 [HIGH]: **Idempotency replay attack to suppress a legitimate retry** — an attacker who learns a victim's Idempotency-Key (e.g. via shoulder-surfing R1 gestures or network observation in a non-HTTPS dev environment) could POST a malicious body with that key BEFORE the victim's legitimate retry. RFC behavior: same key + different body → 422, which means the attacker's body is rejected but the victim's later retry with the SAME body as their original would succeed (the cache wasn't populated). Real attack: attacker POSTs identical body → caches a `phase-07-pending` stub response → victim's retry receives the cached stub instead of triggering the real handler. Mitigation: idempotency keys are non-secret per RFC (intent identifiers, not auth tokens) — the bearer token at the route level IS the authentication. Phase 07 will additionally bind the cached response to the bearer token (entry key = `${key}:${bearer-hash}`) but Phase 03 cannot do this yet because the stub response is identical for any bearer. Document this limitation in JSDoc and threat-model the assumption that bearer token possession is the trust root.
T-03-06 [HIGH]: **Idempotency cache memory exhaustion via key flooding** — an attacker with a valid bearer could POST thousands of distinct random keys to grow the `Map<key, entry>` unboundedly within the 60s window. Mitigation: bound the store size at 10_000 entries (well above any realistic single-tenant 60s rate); on insertion when size === 10_000, evict the oldest entry (Map preserves insertion order — call `this.store.delete(this.store.keys().next().value)`). Existing `@fastify/rate-limit` (Phase 02) at 100 req/min per bearer already caps the attainable key count.
T-03-07 [MEDIUM]: **Idempotency-Key logged inadvertently** — Fastify's default request logger may serialize headers in error scenarios. Mitigation: extend `redact` config in `server.ts` to include `'headers.idempotency-key'` and never construct a log message that contains the full key. Where the key MUST appear in debug logs (idempotency miss / hit telemetry), use `key.slice(0, 8) + '...'`.
T-03-08 [LOW]: **Replay of expired entry race** — if the TTL boundary is reached mid-request (e.g. handler takes 59s), the entry stored at `onSend` time may be evicted by a parallel request's `get()` before the handler finishes. Mitigation: store TTL is measured from `cachedAt = Date.now()` at `onSend` insertion; a fresh insert resets the clock. Worst case is a slightly-extended window which is harmless for the dedup semantic.
</threat_model>

- `packages/bridge/src/middleware/idempotency.ts` — new file. Implement `IdempotencyStore`:
  ```typescript
  const TTL_MS = 60_000;
  const MAX_ENTRIES = 10_000;

  export class IdempotencyStore {
    private readonly store = new Map<string, IdempotencyEntry>();

    get(key: string): IdempotencyEntry | undefined {
      const entry = this.store.get(key);
      if (entry === undefined) return undefined;
      if (Date.now() - entry.cachedAt > TTL_MS) {
        this.store.delete(key);
        return undefined;
      }
      return entry;
    }

    set(key: string, entry: Omit<IdempotencyEntry, 'cachedAt'>): void {
      // T-03-06: bound size; Map preserves insertion order → oldest is first key.
      if (this.store.size >= MAX_ENTRIES && !this.store.has(key)) {
        const oldest = this.store.keys().next().value;
        if (oldest !== undefined) this.store.delete(oldest);
      }
      this.store.set(key, { ...entry, cachedAt: Date.now() });
    }

    get size(): number { return this.store.size; }
    clear(): void { this.store.clear(); }
  }
  ```
  And `registerIdempotencyHooks(app, store)`:
  - preHandler:
    - If `request.method !== 'POST'` → return (pass-through for GET/PUT/etc.).
    - If `request.url.startsWith('/internal/')` → return (excluded; T-03-05 + Open Question 4).
    - Read `const rawKey = request.headers['idempotency-key']`. If `typeof rawKey !== 'string' || rawKey.trim() === ''` → return (no key, no dedup).
    - Compute `const bodyHash = createHash('sha256').update(JSON.stringify(request.body ?? null)).digest('hex')`.
    - `const existing = store.get(rawKey);`
    - If `existing` and `existing.requestBodyHash !== bodyHash` → `reply.status(422).send({ error: 'idempotency_key_conflict', message: 'Idempotency-Key was already used with a different request body' })`. **Return** — Fastify short-circuits the handler when `reply.send()` is called from a hook.
    - If `existing` and matching hash → `reply.status(existing.responseStatus).send(existing.responseBody)`. The onSend hook MUST detect this replay path and NOT re-cache (avoid duplicate cache writes); see onSend logic below.
    - Otherwise (no existing entry): `request.idempotencyKey = rawKey; request.idempotencyBodyHash = bodyHash;` — onSend will cache the eventual response.
  - onSend:
    - If `request.idempotencyKey === undefined` → return `payload` (pass-through; this includes the replay case where preHandler short-circuited and reset `request.idempotencyKey` to undefined effectively — see implementation note).
    - Parse the payload back to a value if it's a JSON string: `let parsed: unknown = payload; if (typeof payload === 'string') { try { parsed = JSON.parse(payload); } catch { parsed = payload; } }`.
    - `store.set(request.idempotencyKey, { requestBodyHash: request.idempotencyBodyHash ?? '', responseStatus: reply.statusCode, responseBody: parsed });`
    - Return `payload` unchanged.
  - Implementation note: to ensure the replay path does NOT trigger a redundant onSend cache write, do NOT set `request.idempotencyKey` in the replay branch — only in the "new key" branch. The onSend `if (request.idempotencyKey === undefined) return payload` guard handles this.
