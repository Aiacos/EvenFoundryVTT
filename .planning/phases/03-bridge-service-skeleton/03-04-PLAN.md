<threat_model>
T-03-13 [HIGH]: **Unbounded /v1/tools/:name URL parameter** — `POST /v1/tools/<arbitrary>` accepts any name string. Without explicit whitelist the handler dispatches via `TOOL_DISPATCH_TABLE[name]` which returns `undefined` for unknown names but still consumes a Fastify request, populates metrics, and emits an idempotency cache entry. Mitigation: the handler first checks `if (!(name in TOOL_DISPATCH_TABLE))` and returns 404 BEFORE Zod parse and BEFORE the idempotency cache is touched (preHandler runs first chronologically, but the dispatcher returning 404 short-circuits with `reply.send` before `onSend` caches anything... wait — Plan 03-02 onSend DOES cache 404 responses if Idempotency-Key was set). Decision: return 404 in the handler; the 404 response IS legitimately cacheable per RFC (a key-bound dedup of a known-error response is correct behavior). Acceptable.
T-03-14 [HIGH]: **Phase 07 write-path pre-emption via unstubbed handler** — if Plan 03-04 accidentally calls into a real `activity.use()` (e.g. via the existing socketlib `evf.getCharacterSnapshot` chain), it would violate Phase 02's read-only contract and the single-workflow-origin discipline Option A (Phase 0 D-15). Mitigation: the 7 new socketlib handlers in `socketlib-handlers.ts` return ONLY the literal `{ status: 'phase-07-pending' }` object — they do NOT import from `dnd5e`, do NOT call any `game.actors.get(...).update(...)`, do NOT touch `MidiQOL`. Verified by grep: `grep -cE "(activity\\.use|completeActivityUse|\\.update\\(|MidiQOL)" packages/foundry-module/src/pair/socketlib-handlers.ts` MUST return 0 net-new mentions (existing Plan 02/05 lines are unaffected; new handlers add zero).
T-03-15 [MEDIUM]: **JSON Schema drift from Zod schema** — if a Phase 11 foundry-mcp client reads `inputSchema` from `GET /v1/tools` and Phase 07 silently changes the Zod schema without re-running `.toJSONSchema()`, the discovery contract drifts. Mitigation: TOOL_REGISTRY computes `.toJSONSchema()` at module-load time (not request time) — every server boot recomputes from the live Zod definition. Bonus: a test in `tools.test.ts` deep-equals the registry entry's `inputSchema` against `Schema.toJSONSchema()` to detect manual drift.
T-03-16 [LOW]: **Tool name leakage in logs** — if a 404 path logs the requested name, attackers fingerprint the registry. Acceptable: the canonical list is already public via `GET /v1/tools` (bearer-auth gated). The 404 leaks nothing not already discoverable.
</threat_model>

<!-- POST /v1/tools/:name handler flow (this plan): -->
```typescript
app.post<{ Params: { name: string } }>('/v1/tools/:name', async (request, reply) => {
  const { name } = request.params;
  // T-03-13: 404 before any auth check is acceptable (no info leak; list is public via GET /v1/tools)
  if (!(name in TOOL_DISPATCH_TABLE)) {
    return reply.status(404).send({ error: 'unknown_tool', tool: name });
  }
  // Bearer auth
  const authHeader = request.headers.authorization;
  if (!authHeader?.startsWith('Bearer ')) return reply.status(401).send({ error: 'invalid_token' });
  const token = authHeader.slice('Bearer '.length);
  const validation = await tokenCache.validate(token);
  if (!validation.valid) {
    if (validation.reason === 'foundry_unreachable') return reply.status(503).send({ error: 'foundry_unreachable' });
    return reply.status(401).send({ error: 'invalid_token' });
  }
  // Body validation
  const schema = TOOL_INPUT_SCHEMAS[name as ToolName];
  const parsed = schema.safeParse(request.body);
  if (!parsed.success) return reply.status(400).send({ error: 'invalid_body', details: parsed.error.issues });
  // Dispatch (idempotency middleware from Plan 03-02 wraps this — replay returns cached response BEFORE we reach here)
  const idempotencyKey = request.headers['idempotency-key'];
  const result = await TOOL_DISPATCH_TABLE[name as ToolName](parsed.data, typeof idempotencyKey === 'string' ? idempotencyKey : undefined);
  return reply.status(200).send(result);
});
```
