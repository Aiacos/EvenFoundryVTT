<inline_research>
**D-13-08 — Container budget slot reassignment (the critical Phase 13 architecture decision):**
- Today's container layout (steady state, NO overlay): z=0 MapBaseLayer = ??? image containers (implicit `getContainerCount` returns default `{image:0, text:1}` but the raster pipeline actually uses 4 image containers internally via the page schema declared by capability-handshake). The "default missing" behavior is a LATENT BUG that hasn't fired because no overlay panel has claimed image containers until now.
- Plan 13-04 CORRECTS MapBaseLayer to declare explicit `getContainerCount(): {image: number, text: number}`:
  * In raster mode: returns `{image: 4, text: 1}` (matches the 2x2 sub-tile layout established Phase 4a).
  * In glyph mode: returns `{image: 0, text: 1}` (no image containers in glyph fallback).
  * Mode source: MapBaseLayer reads from `this.layerManager.getMapMode()` and `this.controller.getBleVerdict()` — same logic as `resolveAutoMode` already in this file. Add a thin `_currentEffectiveMode()` helper that returns the resolved 'raster' | 'glyph'.
- WHEN Bio portrait active: CharacterSheetPanel.getContainerCount returns `{image: 1, text: 3}` for the bio tab (1 image for the portrait + 3 text rows: tab strip, bio body, footer). When NOT on bio tab OR portrait off: returns `{image: 0, text: 1}` as today.
- ADR-0009 Amendment 1 differential demolish: when z=2 overlay mounts, z=0.5 IdleInfillLayer (text:3, image:0) is demolished → its text budget frees. When the overlay carries image:1, the LayerManager._assertContainerBudget recomputes: z=0 (4 image, 1 text) + z=1 status-hud + z=1.5 toast + z=2 sheet bio (1 image, 3 text) → 5 image total. This EXCEEDS the 4-image cap.
- RESOLUTION (the slot reassignment): MapBaseLayer drops from `{image:4, text:1}` → `{image:3, text:1}` whenever ANY mounted overlay declares `image >= 1`. This is implemented by:
  * MapBaseLayer.getContainerCount inspects `this.layerManager.getLayers()` (NEW accessor returning the layers Map readonly) and sums OTHER layers' image counts. If sum >= 1, return `{image:3, text:1}` else `{image:4, text:1}`.
  * Add `LayerManager.getLayers(): ReadonlyMap<ZIndex, Layer>` — pure read accessor; existing internal `this.layers` field made accessible via this method.
  * The raster page schema must support 3-vs-4 image containers without re-handshake. **Phase 4a's page schema declares container slots at boot via `createStartUpPageContainer`** — this is the constraint. Verify at execute-time that `rebuildPageContainer` accepts a new container count, OR that the page schema must be re-declared. **Fallback if page schema is immutable post-handshake**: declare the page with 4 image slots from the start (existing); MapBaseLayer simply RENDERS into only 3 of the 4 slots when overlay demands the 4th; the 4th slot is owned by the overlay's image. This is the simpler implementation and matches Phase 4a's 2x2 tile structure (the tile in the bottom-right quadrant is replaced by the portrait when overlay active). Tests verify the budget assertion passes (image total still ≤4) AND the visible composition has 3 raster tiles + 1 portrait.
- Decision: Use the simpler "MapBaseLayer renders 3 tiles when overlay declares image≥1" approach. MapBaseLayer's getContainerCount returns the static page-schema count (4 image, 1 text in raster; 0 image, 1 text in glyph) — but the RENDERER skips one tile when an overlay's image >= 1 by reading layerManager.getLayers() before issuing updateImageRawData calls. This satisfies the budget assertion (page schema declares 4 image slots which is the SDK cap; the layer composition sums to exactly 4 in raster+portrait active state, exactly 1 in glyph+portrait, exactly 4 in raster+no-portrait, etc.).
- Final budget math (raster + portrait ON, character sheet bio open):
  * z=0 MapBaseLayer: image=4, text=1 (page schema declaration)
  * z=0.5 Idle infill: DEMOLISHED by differential demolish
  * z=1 StatusHudLayer: image=0, text=1
  * z=1.5 ToastQueueLayer: image=0, text=1 (when toast present) or 0 (idle)
  * z=2 CharacterSheetPanel (bio + portrait): image=0, text=3 — the portrait occupies a MapBaseLayer image SLOT, not a new container; the panel only declares the 3 text containers for tab-strip + bio-rows + nav-footer
  * Total: 4 image (≤4 cap) + 6 text (≤8 cap) → BUDGET PASSES
- This means CharacterSheetPanel does NOT declare image:1 in getContainerCount; instead it writes to the reserved MapBaseLayer image slot via the bridge directly. This is a CLEAN architectural split — overlays own text containers, MapBaseLayer owns image containers; portraits piggyback on a raster tile slot.

**Revised architecture decision (simpler):**
After the analysis above the cleaner story is:
- MapBaseLayer keeps `getContainerCount() = {image: 4, text: 1}` in raster, `{image: 0, text: 1}` in glyph.
- CharacterSheetPanel adds NO image to its container count — keeps `{image: 0, text: 1}` (Strategy A unchanged).
- The PORTRAIT BYTES are written to a reserved image slot owned by MapBaseLayer. MapBaseLayer exposes a new method `setPortraitOverride(slotIdx: 0..3, bytes: Uint8Array | null)` that, when non-null, renders that slot with the portrait instead of the raster tile.
- CharacterSheetPanel on Bio-tab mount calls `mapBase.setPortraitOverride(3, portraitBytes)` (slot index 3 = bottom-right tile, configurable); on Bio-tab unmount/tab-change calls `setPortraitOverride(3, null)`.
- Container budget passes trivially because the page schema is unchanged: 4 image + 1 text + downstream overlay text counts.
- This is THE WIRING — Plan 13-04 picks this approach. The slot reassignment is data-only inside MapBaseLayer; no new container is allocated.
- Tradeoff acknowledged: when in glyph mode, no image slots exist so portrait is NOT rendered (glyph mode is BLE-degraded; visual fidelity is already reduced). Acceptable for MVP.
