<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| bridge → g2-app (WS) | Inner ActionEconomyPayload safeParse is the second of two trust boundaries (outer EnvelopeSchema + inner payload). |
| g2-app cache → panel | `action-economy-state` is in-process; ActionOptionsModal trusts cache contents because the dispatcher already validated. |
| panel → bridge (tool.invoke emit) | Preconditioner short-circuit is fail-open: when cache is empty (`null`), modal emits as normal (server-side validation is authoritative). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Tampering | Client cache stale or manipulated | mitigate | Preconditioner is FAIL-OPEN: when `getActionEconomyState(actorId) === null`, the tap proceeds normally (AOM-PRE-04). Server-side `combat-action-tracker.ts` is the source of truth. Cache cannot create a new false-action-used state; at worst it leaves the user to find out via the result toast (legitimate). |
| T-09-02 | Spoofing | T-09-MOD-01 widget mis-render | mitigate | INV-1 fixtures (SHR-EW-06) verify character-perfect output; `_actionEconomy: null` branch preserves existing fixtures (no regressions). |
| T-09-03 | Information disclosure | error toast leaks cross-player state | mitigate | Toast `id` includes `actorId` — only the user's own actor produces toasts because the dispatcher already filtered by `recipientUserId === currentUserId` upstream (Plan 09-01 AED-05). |
| T-09-04 | Denial of service | repeated taps spam error toasts | accept | `Toast.id` is per-tap-timestamp — ToastQueueLayer FIFO+squash already handles overflow (Phase 4b TOAST-01). |
</threat_model>
