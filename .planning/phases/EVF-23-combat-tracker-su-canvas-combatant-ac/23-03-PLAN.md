<objective>
Create `CanvasCombatTrackerPanel` — the second z=2 canvas overlay panel — as a direct
structural mirror of `CanvasCharacterSheetPanel` (dual-interface CanvasLayer + OverlayPanel,
`_scrollOffset`, dirty-gate, chrome pre-bake, idempotent gesture/sub unsubscribe). It reuses
the glyph tracker's tested business logic (`computeWindow`, `renderCombatTrackerContent`,
`renderCombatantRow`) via Approach A (glyph strings drawn with `ctx.fillText` per row), adds a
full-contrast current-turn highlight (inverted fill), self-subscribes to `combat.turn` +
`combat.state` for live updates with auto-follow on turn change (D-23.3), renders real `ac`
(with `' --'` fallback preserved), and wires the boot dispatch gate + handler injection.

This is the third atomic piece (D-23.6) — it depends on Plan 01 (schema `ac`) and Plan 02
(reader `ac`). Implements RCOMB-01 (canvas combat tracker) and the render half of RDATA-05.

Per D-23.5: `PanelGestureBus` and `panel-router.ts` are NOT modified — routing reuses the
existing boot dispatch gate + setPanelInstanceHandler mechanisms (same as Phase 21's
character-sheet → canvas-character-sheet). A grep guard asserts both files are byte-unchanged.

Purpose: Players see the combat tracker as a raster overlay with real AC, current-turn
highlight, and 5-combatant scroll, with gesture parity to the glyph version.
Output: `CanvasCombatTrackerPanel` class + panel id `'canvas-combat-tracker'` + boot arm +
combat.turn/combat.state subscription wiring + updated glyph panel AC field.
</objective>
