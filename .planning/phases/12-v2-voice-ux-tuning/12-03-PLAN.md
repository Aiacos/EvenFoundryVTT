**Research integrated inline:**

1. **Deepgram Nova-3 Multilingual streaming endpoint** — `wss://api.deepgram.com/v1/listen` (Deepgram-canonical streaming URL; documented at developers.deepgram.com/reference/listen-live and used by `deepgram-node-sdk`). Query parameters for Phase 12: `model=nova-3` (per CONTEXT.md D-12-01), `language=multi` (Nova-3 Multilingual code-switching mode), `punctuate=true` (CONTEXT.md decision), plus the linear16 / 16kHz / mono triplet `encoding=linear16&sample_rate=16000&channels=1` REQUIRED when sending raw PCM frames (Deepgram default is opus; raw s16le PCM must be declared explicitly per Deepgram streaming-codecs docs). The Even Hub SDK delivers PCM as Uint8Array per the `AudioEventPayload` type — that maps directly to `linear16` on the wire with no transcoding (Specs.md §3.5 already commits 16 kHz s16le mono).

2. **Deepgram auth header** — `Authorization: Token <DEEPGRAM_API_KEY>`, NOT `Bearer <KEY>`. Verified via `deepgram-node-sdk` README ("the SDK uses `Authorization: token DEEPGRAM_API_KEY` format"). The lowercase `Token` (vs `Bearer`) is the canonical Deepgram convention — bridge implementation MUST use `Token` exactly. Test will assert the literal header string.

3. **Deepgram Results frame shape** — JSON over WS: `{ type: 'Results', channel: { alternatives: [{ transcript: string, confidence: number, words: [...] }] }, is_final: boolean, speech_final: boolean, start: number, duration: number }`. Bridge parses this into VoiceTranscriptPayload by taking `channel.alternatives[0].transcript` + `channel.alternatives[0].confidence` + `is_final` + a constant `language: 'multi'` (Nova-3 Multilingual doesn't surface per-utterance language attribution at the Results envelope layer — the LLM downstream handles code-switching from the transcript text itself per Plan 12-02 worked examples).

4. **PCM 16 kHz s16le mono frame format** — Even Hub SDK delivers `event.audioEvent.audioPcm: Uint8Array` (per `node_modules/@evenrealities/even_hub_sdk/dist/index.d.ts` line 860). The byte order is little-endian signed 16-bit (s16le) at 16 kHz mono per Specs.md §3.5 verbatim. Frame size is whatever the SDK delivers per `onEvenHubEvent` fire; bridge forwards verbatim without re-chunking. Deepgram accepts arbitrary frame sizes for `linear16` streaming.

5. **`bridge.audioControl(true|false)` SDK shape** — Verified against the installed `@evenrealities/even_hub_sdk@0.0.10`: `audioControl(isOpen: boolean): Promise<boolean>` (line 1187 of `index.d.ts`). The boolean argument toggles G2 mic state (true = capture on, false = off). Phase 12 calls `EvenAppBridge.getInstance().audioControl(true)` at audio-capture start and `audioControl(false)` at stop. The existing `hub-polyfill.ts` does NOT cover audioControl (Phase 2 polyfill is scoped to setItem/getItem/eventBus only) — Phase 12 audio-capture goes DIRECTLY against `EvenAppBridge.getInstance()` per the hub-polyfill JSDoc "When NOT to use this — In Phase 4a+ code that writes directly against `EvenAppBridge` — use the SDK API directly".

6. **Event subscription shape** — `EvenAppBridge.getInstance().onEvenHubEvent((event: EvenHubEvent) => { if (event.audioEvent) { /* event.audioEvent.audioPcm: Uint8Array */ } })`. The onEvenHubEvent return value is the unsubscribe function (per d.ts line 1266 `(event: EvenHubEvent) => () => void`). Audio-capture stores it on the handle so stop tears down both the audioControl(false) call and the event subscription.

7. **Bridge route placement** — `registerAudioStreamRoute(app, ...)` runs at step 10 in `server.ts` (after the existing `/ws` handshake registration at step 9). It mounts a separate Fastify route `/v1/audio/stream` so the existing `/ws` envelope-protocol path is untouched. Both routes share the same DeltaEmitter (the audio path emits VoiceTranscript envelopes that other `/ws` clients can subscribe to — useful for diagnostics and for the Phase 11 MCP server's WS subscription path to surface transcripts back to Claude Desktop as resource updates if needed in V3).

8. **Token redaction (T-12-02)** — `packages/bridge/src/server.ts` already has a pino redact list (line 122-132). This plan extends it with the field paths `deepgramKey`, `apiKey`, and `*.DEEPGRAM_API_KEY` so the Deepgram key never leaks into logs. The voice-secret-redact.test.ts asserts that logging a structured object containing all 4 candidate field names (`apiKey`, `deepgramKey`, `headers.authorization`, `DEEPGRAM_API_KEY`) produces a stringified log line with zero `'sk-fake'` substring matches.

9. **Bridge-soft-fail (no key set)** — Mirrors Phase 11 11-04 BridgeClient soft-fail pattern: bridge boots unconditionally; if `DEEPGRAM_API_KEY` is unset, `createDeepgramStt({ apiKey: undefined })` returns a stub whose `connect()` always returns a closed-stream marker; `audio-stream-route` checks `deepgramStt.isEnabled()` and closes incoming WS with code 1011 + reason `voice-disabled` when disabled. This makes Phase 12 backward-compatible with the MVP — operators who don't enable voice keep getting a working bridge.

10. **No new socketlib handlers** — Phase 12 touches NOTHING in `packages/foundry-module/`. The 14-socketlib-handler invariant (Phase 7+8+9+10+11 closure assertions) holds unchanged. Phase 12 closure asserts this by grepping `registerComplexHandler` count at exactly 14.
