<objective>
Add the hot-update refresh path so the Deepgram session's keyterm list stays current when the Foundry compendium changes. The trigger flow is:

```
Foundry compendium delta
  → entity-pack-reader.ts (module)
  → POST /internal/delta with r1.entities.available envelope
  → handleEntityPackEnvelope (step 8 multiplex)
  → EntityPackCache.set(payload)
  → [NEW] EntityPackCache fires onChange listener
  → [NEW] KeytermRefresher.scheduleRefresh()  ← debounced 250ms, mutex-serialised
  → [NEW] DeepgramAdapter.refreshKeyterm()    ← re-emits keyterms for active sessions
```

Two implementation notes locked by CONTEXT.md:

- **Debounce 250ms (CONTEXT D-07)** to coalesce multi-delta bursts (e.g. user installs a compendium pack that triggers 100+ updateCompendium hooks within 200ms). Use `setTimeout` (RxJS would inflate deps; not needed).
- **Mutex serialisation (CONTEXT D-07)** to prevent overlapping in-flight refreshes. Use a Promise-queue flag pattern (no third-party lib).

The Deepgram WS protocol does not support mid-stream keyterm hot-swap. The realistic refresh strategy is: **next connect() picks up the new keyterm list** (already true thanks to plan 15-02 lazy keytermProvider). The `refreshKeyterm()` method on the adapter is therefore an **invalidation signal**, not a wire-level re-config — it logs that the next connect will use updated keyterms. The "≤ 5 minutes" SLA from VOICE-09 is naturally satisfied because Deepgram sessions are short-lived (per-utterance reconnects) — a fresh connection within 5 minutes is the norm.

For Phase 15 software-only scope, the refresher's job is therefore:
1. Receive the EntityPackCache change notification
2. Debounce 250ms
3. Acquire the mutex
4. Compute the new keyterm list via the merger (does it differ from the last-emitted list?)
5. If yes, log a structured `keyterm.refreshed` event (pino logger) and notify the adapter so subsequent connect() calls use the fresh values
6. Release the mutex

VOICE-09 success criterion is **observable in tests**: a Vitest test simulates `cache.set(updatedPayload)` and asserts (a) the refresher's listener is invoked, (b) the merger output reflects the new entries, (c) a `keyterm.refreshed` log event fires, (d) burst of 5 set() calls within 250ms results in exactly 1 log event.

</objective>
