| foundry-mcp → bridge | Trusted (same homelab); bearer forwarded; bridge re-validates via Phase 7 token-cache |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Information disclosure | env.ts BootError + pino logger | mitigate | BootError.message never includes bearer value; pino redact list covers bearer/token/Authorization/EVF_BEARER (verified by test 7 in Task 1) |
| T-11-02 | Spoofing | HTTP entry Authorization header | mitigate | Constant-time bearer compare (crypto.timingSafeEqual on equal-length Buffers; length-fast-fail on mismatch); 401 on mismatch BEFORE invoking transport.handleRequest |
| T-11-03 | Tampering | Streamable HTTP body | mitigate | SDK validates JSON-RPC envelope structure via Zod internally; we never trust client-supplied JSON for routing decisions in this plan (tool-level validation lands in 11-02) |
| T-11-04 | Repudiation | All transport ops | accept | MVP single-tenant homelab; audit log lives on the bridge side via existing Phase 7 audit-log. MCP-side request log is pino-stdout (HTTP) / pino-stderr (stdio) — sufficient for V2 |
| T-11-05 | Denial of service | HTTP entry without rate-limit | accept | Phase 11 OPZIONALE; bridge re-validates every request and has its own rate-limit (Phase 3). Adding rate-limit here is V3+ scope per CONTEXT deferred list |
</threat_model>
