    2. **Geometry (INV-2, verified 2026-06-05):** image containers max 4 per page, each 20–200px wide × 20–100px tall → raster surface = **400×200 (4 tiles 200×100), NOT 576×288**. `HUD_TILE_GEOMETRY` = 200×100. On-screen placement of the 400×200 region inside 576×288 is **parameterized** (default deferred to Phase 20). This **supersedes the original ADR-0013 Decision** statement of 288×144 / full-screen tiles — state explicitly that the 288×144 PoC geometry passed only in the simulator (which does not enforce hardware image-size limits) and is rejected.
    3. **Capture-container re-mapping:** the 5th container `'hud-capture'` is a **full-screen text container** with `isEventCapture:1`, behind the 4 image tiles (NOT a zero-size container — undocumented). Cite the EvenHub first-app example using a full-screen text container as the capture target. INV-5 preserved; gesture routing unchanged.
    4. **Container budget fixed mode + serialized push:** in canvas mode the budget is FIXED at page creation (5 containers: 4 image + 1 capture text). `CanvasLayer.getContainerCount()` returns `{image:0, text:0}` (canvas layers don't allocate per-container). `_assertContainerBudget()` bypasses the per-layer sum in canvas mode. `updateImageRawData` does NOT allow concurrent sends → `_compositeAndPush()` serializes the 4 tile pushes (await each). Schema is fixed: panel change (Phase 21+) = `updateImageRawData` on existing tiles, NOT `rebuildPageContainer` (which flickers / loses state).
    5. **Glyph fallback + `_flushPage()` schema selector:** `LayerManager.renderMode: 'canvas' | 'glyph'`; glyph path byte-identical to today (BLE-degraded, ADR-0005 Branch A); mode switch atomic via `bundle([])`. `_flushPage()` selects `buildHudRasterPageSchema()` (5 containers) when `renderMode === 'canvas'`, else `buildStatusViewTextContainers()` (3 containers) when `renderMode === 'glyph'`.
