# Quick Task 260513-l12: Fix ApplicationV2 ReferenceError (Foundry v13+ namespace)

## Bug

`evenfoundryvtt@0.1.2` crashed at module load time in Foundry v13/v14 with
`Uncaught (in promise) ReferenceError: ApplicationV2 is not defined`. The root
cause: `PairModal.ts` contained `export class PairModal extends ApplicationV2`
where `ApplicationV2` was resolved as a bare global. Foundry v13 relocated this
class to `foundry.applications.api.ApplicationV2` — the bare global no longer
exists at module evaluation time. Because `class PairModal extends ApplicationV2`
is evaluated when `dist/module.js` is first parsed (not deferred to function
call time), the entire `init` hook chain aborted before
`game.settings.register('evenfoundryvtt', 'bearerRegistry', ...)` could run.
Manual console invocation of `game.settings.register(...)` succeeded, confirming
the API was live — the problem was purely the class-body evaluation order.

## Fix

Three coupled changes, committed atomically per INV-4 (no half-updated state):

- **Source**: `src/pair/PairModal.ts` — added
  `const { ApplicationV2 } = foundry.applications.api;` immediately after the
  import block. The `extends ApplicationV2` on line 190 and
  `...ApplicationV2.defaultOptions` on line 216 are unchanged; they now bind
  to the locally destructured variable instead of the missing bare global.
- **Types**: `src/types/foundry-globals.d.ts` — replaced the false-global
  `declare class ApplicationV2 { ... }` with
  `declare namespace foundry { namespace applications { namespace api { class ApplicationV2{...} } } }`.
  Also removed the unused `declare class Application { ... }` (zero dead code,
  INV-4). The namespace declaration merges cleanly — no other `declare namespace
  foundry` existed in the package.
- **Tests**: 4 test files updated — every `vi.stubGlobal('ApplicationV2',
  ApplicationV2Stub)` call (19 total: 6 + 7 + 5 + 1) replaced with
  `vi.stubGlobal('foundry', { applications: { api: { ApplicationV2:
  ApplicationV2Stub } } })`. Local `ApplicationV2Stub` class bodies are
  unchanged — only the registration mechanism changed. The neighbouring
  `vi.stubGlobal('Application', ApplicationStub)` calls were left in place
  (out of scope, harmless, removing them risks regression).
