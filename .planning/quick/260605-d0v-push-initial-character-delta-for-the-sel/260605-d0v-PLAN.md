<objective>
On a new g2-app WS client connection, the bridge proactively pushes an initial `character.delta` (full CharacterSnapshot) for the currently-selected actor, so the glasses HUD renders real character data immediately on connect instead of staying blank until the next Foundry-triggered delta.

Purpose: Closes the documented gap (MEMORY: "g2app-hud-launch-chain-status" — engine boots in sim but glasses render is blank for lack of a WS data push on connect). The roster push path (`CharacterListCache`, Quick Task 260604-eyf) already tells the bridge which actors exist; this task uses the first roster entry as the selected actor and pushes its full snapshot through the existing `getCharacterSnapshot` path.

Output:
- `delta-emitter.ts` gains a targeted `sendInitialToSession(sessionId, type, payload)` (single-session send, same envelope/replay/lastSeq semantics as `emitDelta`).
- New `initial-snapshot.ts` with `pushInitialCharacterDelta(...)`: selects actorId from `CharacterListCache`, fetches+validates the snapshot via `foundryFn`, and sends `character.delta` to the new session. Graceful no-op on cold roster / null snapshot / schema mismatch.
- `server.ts` calls `pushInitialCharacterDelta` inside the `/ws` handshake `.then()`, right after `deltaEmitter.registerSession(...)`.
- VERIFY: Vitest unit + integration tests assert the bridge emits `character.delta` on WS connect with a MOCK CharacterSnapshot payload (and emits nothing on cold roster). Optional sim smoke check documented as a manual step.
</objective>
