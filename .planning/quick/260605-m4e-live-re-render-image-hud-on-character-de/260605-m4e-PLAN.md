<objective>
Convert the raster-HUD PoC from a single-frame-on-connect draw into a LIVE re-render loop:
when a new `character.delta` snapshot arrives over the existing WS data path, redraw the HUD
canvas, rebuild the 4 tiles, and re-push them via `updateImageRawData`. This closes TODO-hud-raster
item #1 (ADR-0013 §Scope step 3, "wire `character.delta` → redraw → tiles").

This is the SIMPLE re-render wiring only. On each snapshot, redraw + re-push ALL 4 tiles. The
~5fps debounced `RasterController` delta loop with xxhash sub-tile diffing is TODO #2 — a SEPARATE
later task — and is explicitly NOT in scope here.

Purpose: the image-HUD must track the live character state, not freeze on the first frame.
Output:
- `packages/g2-app/src/hud/hud-live-render.ts` — pure render-once orchestrator (unit-tested).
- Extended `boot-hud-raster-poc.ts` — WS + event bus + `character.delta` subscription → live loop.
- Live-sim visual gate proving the HUD re-renders when the underlying snapshot changes.
</objective>

<key_findings>
Read-the-code findings the executor MUST rely on (do NOT re-derive — they were verified during planning):

1. WHERE snapshots come from: `createWsEventBus(ws, seqTracker?)` in
   `packages/g2-app/src/internal/boot-engine-core.ts` (exported). It attaches ONE persistent
   `ws.addEventListener('message', …)` and returns `{ subscribe(channel, fn), setPerfProbe(p) }`.
   `subscribe('character.delta', fn)` invokes `fn(payload)` once synchronously with the cached
   last value (last-value replay — this is what delivers the on-connect snapshot) and then on every
   future `character.delta` envelope. THIS is the exact mechanism `StatusHudLayer` uses
   (`status-hud-layer.ts#subscribeWsEvents` → `subscribe('character.delta', raw => this._onDelta(raw))`).

2. PAYLOAD SHAPE: the `character.delta` payload is the `CharacterSnapshot` object directly.
   `StatusHudLayer._onDelta` validates it with `CharacterSnapshotSchema.safeParse(raw)` and on
   failure does `console.warn` + ignore (no throw). MIRROR this gate exactly (it is also the
   T-ksd-01 mitigation already used in the PoC's `fetchSnapshot`).

3. HOW the PoC fetches today: `boot-hud-raster-poc.ts#fetchSnapshot` does a one-shot REST
   `GET {bridgeUrl}/v1/character/{characterId}` then `renderHudFrame → buildHudTiles → pushHudTiles`
   ONCE. The conversion: keep the one-shot REST fetch as the FIRST frame (so the HUD draws
   immediately even before the first WS delta — preserves current behavior), THEN open a WS and
   subscribe to `character.delta` so every subsequent snapshot re-invokes render→assemble→push.
   Because of last-value replay, the WS subscription will also fire immediately if the bridge has
   already cached an on-connect delta — that is fine (it just redraws the same/fresher data; this
   naive "redraw on every snapshot" is the agreed scope).

4. WS URL derivation: use `toWsConnectUrl(opts.bridgeUrl)` (exported from
   `packages/g2-app/src/engine/ws-url.ts`) — the bridge serves its socket at `/ws` and requires a
   `ws`/`wss` scheme. Wait for OPEN before subscribing (mirror `awaitWsOpen` in boot-engine-core —
   a small local `readyState === 1` / `addEventListener('open')` wait is fine; do NOT export the
   private one).

5. NO HANDSHAKE NEEDED for the no-auth dev/sim path the PoC targets: the bridge is booted with
   `EVF_DEV_NO_AUTH` and pushes the on-connect `character.delta` for the seeded actor without a
   capability handshake. If the WS connection or subscription fails, the PoC MUST stay alive showing
   the REST first-frame (fail-soft) — a WS failure must NOT white-screen or abort the boot.
   (If during execution you discover the no-auth bridge requires a minimal handshake to start
   emitting deltas for `characterId`, send the same minimal handshake `bootEngine` uses via
   `performCapabilityHandshake(ws, 'dev-no-auth', locale, undefined, 10_000, characterId)` — but
   verify against the running bridge before adding it; prefer the no-handshake path if deltas flow
   without it. Whichever path: keep it fail-soft.)

6. SIM SNAPSHOT-CHANGE TRIGGER: edit a fixture (e.g. `scripts/sim-fixtures/character-artemis.json`,
   change `hp`) and run `pnpm sim seed` (re-POSTs to `POST /internal/delta`); the bridge then emits
   a fresh `character.delta` over the live WS → the subscription fires → HUD re-renders. This is the
   definitive visual gate (canvas text is NOT unit-testable in happy-dom, exactly as PoC 260605-ksd).
</key_findings>
