<objective>
Fix the leading-edge throttle in `HudDeltaDriver` so delivered HUD fps rises from ~17 to ≥25 (target 24-30 stable) under continuous ~30fps frame input, without regressing zero-push-on-idle (D-24.3) or stop() teardown.

Purpose: Today the real period is `interval + cycleTime` because the next cycle is only armed by the NEXT event arriving AFTER the timer fires. A trailing-edge re-arm makes the period `max(interval, cycleTime)` — at `minRedrawIntervalMs=33` and ~15ms cycle cost this lands ≥25 fps.

Output: Reworked `_schedule()` / cycle-completion logic in `hud-delta-driver.ts`, extended vitest coverage, and a patch changeset.

  <files>packages/g2-app/src/engine/hud-delta-driver.ts</files>
  <behavior>
    Diagnosis (measured live, do NOT re-derive): leading-edge throttle drops events while `_timer !== null`; the timer fires after the interval, runs `_runCycle` (~15ms main + worker round trip), and the NEXT cycle is only armed by the NEXT event AFTER the fire → real period ≈ interval + cycleTime ≈ 48-60ms → ~17 fps delivered. Fix is trailing-edge re-arm at fixed cadence.

    Required new semantics (mirror the planning_context spec exactly):
    - Add a private `_pendingAgain: boolean = false` flag and a private `_cycleInFlight: boolean = false` flag.
    - `_schedule()`: if a timer is pending (`_timer !== null`) OR a cycle is in flight (`_cycleInFlight`) → set `_pendingAgain = true` and return. Otherwise arm the timer (current behaviour) for `minRedrawIntervalMs`.
    - When the timer fires: clear `_timer`, record a `fireStart = Date.now()` timestamp, set `_cycleInFlight = true`, and run exactly ONE `_runCycle()` (never overlapping). The existing `.catch(...)` console.warn on `_runCycle` rejection must be preserved.
    - On cycle completion (a `.finally(...)` on the `_runCycle()` promise): set `_cycleInFlight = false`; if `_pendingAgain` is set → clear it and arm the next timer for `Math.max(0, minRedrawIntervalMs - (Date.now() - fireStart))` so the period becomes `max(interval, cycleTime)` and NEVER `interval + cycleTime`. If `_pendingAgain` is NOT set → do nothing (idle, no re-arm).
    - `stop()`: clear `_timer` (existing) AND set `_pendingAgain = false` so no cycle fires after stop, even if a cycle was in flight when stop() was called (the finally must NOT re-arm because the flag is cleared). Leave `_cycleInFlight` consistent.

    Behaviour invariants the tests in Task 2 will assert:
    - Test A (cadence): continuous deltas every 50ms for ~1000ms with each cycle changing tile-0 → fires roughly every max(interval, cycleTime); composites delivered ≈ input cap (≥ the count the old DL-08 anti-starvation test expected, and now strictly trailing-paced).
    - Test B (no event loss): one event fired DURING a pending window (timer armed, before fire) → exactly one follow-up cycle after the in-flight one completes.
    - Test C (no overlap): when cycleTime > interval, cycles never overlap and the period collapses to cycleTime (single-in-flight guard holds).
    - Test D (idle): no events during the window → no re-arm, zero pushes (D-24.3 unchanged).
    - Test E (stop during pending): stop() while `_pendingAgain` is set cancels the follow-up — no cycle after stop.
  </behavior>
  <action>
Rework the throttle in `_schedule()` and the cycle lifecycle per the behavior block. Add private fields `_pendingAgain` (boolean) and `_cycleInFlight` (boolean) alongside `_timer`. In `_schedule()`, gate on `this._timer !== null || this._cycleInFlight` to set `_pendingAgain = true` and return; otherwise arm the timer. Extract the cycle launch into a single private path (e.g. an internal `_fireCycle()` or inline in the timer callback) that: clears `_timer`, captures `fireStart`, sets `_cycleInFlight = true`, calls `this._runCycle()`, attaches the existing `.catch` warn, and attaches a `.finally` that sets `_cycleInFlight = false` and, if `_pendingAgain`, clears it and re-arms with `setTimeout(..., Math.max(0, this._opts.minRedrawIntervalMs - (Date.now() - fireStart)))`. Use `Date.now()` (consistent with the existing `getFps`/`_pushTimes` use) — works under vitest fake timers.

In `stop()`, additionally set `this._pendingAgain = false` (keep the existing `clearTimeout` + unsub logic). Ensure the `.finally` re-arm checks `_pendingAgain` AFTER stop could have cleared it, so a stop() during an in-flight cycle prevents the follow-up.

Rewrite the large "THROTTLE, not debounce" JSDoc comment block inside `_schedule()` to document the new TRAILING-EDGE re-arm semantics: period = max(interval, cycleTime), single cycle in flight, `_pendingAgain` coalesces all events during the window into exactly one follow-up, and idle produces no re-arm.
