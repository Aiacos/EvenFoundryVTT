<objective>
Module Foundry v0.1.15 — stream mappa più veloce e leggero. Introduce un nuovo wire format `frame_png` (PNG greyscale lossless, ~1-5KB vs 427KB RGBA), cadenza di cattura configurabile dal DM, throttle leading+trailing sui hook (così il pan continuo emette ~max-wait fps invece di mai), e skip dei frame identici via hash. Decode lato g2-app con back-compat completa per `frame_pixels`.

Purpose: ridurre drasticamente la banda BLE/WS del map stream e aumentare il frame rate percepito sul G2, senza rompere i moduli più vecchi che ancora emettono `frame_pixels`.
Output: nuovo schema `frame_png` in shared-protocol, pipeline di encode nel foundry-module, decode nel g2-app, test bridge pass-through, changesets, bump v0.1.15.

**CRITICAL API CORRECTION (verified empirically against upng-js@2.1.0, 2026-06-11):**
The requirement specified `UPNG.encodeLL([luma.buffer], w, h, 1, 0, 8)`. **`encodeLL` does NOT exist in upng-js 2.1.0** (`typeof UPNG.encodeLL === 'undefined'` — exported members are `encode`, `decode`, `toRGBA8`, `quantize`). Furthermore the palette path (`UPNG.encode(..., 0)` with default `forbidPlte=false`) produces a 4-bit palette PNG (ctype=3) whose `UPNG.toRGBA8` **crashes under Node** (photopea PLTE/tRNS indexing bug). The VERIFIED working recipe that roundtrips luma EXACTLY and stays tiny is:
  `UPNG.encode([rgba.buffer], w, h, 0, /*dels*/ undefined, /*forbidPlte*/ true)`
which yields a lossless ctype=2 RGB PNG (depth 8). Measured: 576×288 gradient → 1.2KB; realistic noisy frame → 3.5KB; encode ~45-110ms; `UPNG.toRGBA8(UPNG.decode(png))` reproduces every source byte exactly. The luma optimization is still applied (compute Rec.601 luma, build an R=G=B=luma RGBA buffer, hash + skip on it) — the wire is greyscale-content, lossless, ~10-300× smaller than RGBA. Use THIS recipe, not `encodeLL`.
</objective>
