<objective>
Fix the socketlib API misuse that crashes the Foundry `ready` hook and blocks real Forge pairing.

ROOT CAUSE (proven via live Forge console export): `registerSocketlibHandlers()` calls
`socketlib.registerComplexHandler(MODULE_ID, id, fn)` — a method that DOES NOT EXIST in the real
`farling42/foundryvtt-socketlib` library. It was invented in the ambient `foundry-globals.d.ts`
declaration (so TS compiled) and mocked in tests (so tests passed), but throws `TypeError:
socketlib.registerComplexHandler is not a function` at runtime. Because `registerSocketlibHandlers()`
is the FIRST call inside `Hooks.once('ready')` in `module.ts`, the throw ABORTS the entire ready
callback BEFORE the HTTP push readers (`registerBearerRegistryReader` / `registerCharacterListReader`,
registered later in the same hook) ever run. Those readers are the `/internal/delta` push path real
pairing depends on — so zero deltas reach the bridge and real pairing is impossible.

This plan:
1. Replaces the fictional API with the real one: `socketlib.registerModule(MODULE_ID)` -> `socket.register(id, fn)`.
2. Moves socketlib handler registration to `Hooks.once('socketlib.ready', ...)` (canonical socketlib timing).
3. Decouples the HTTP push readers + other emitters so they register in `ready` INDEPENDENTLY of socketlib —
   if socketlib is ever absent/broken, the push path STILL works (defense in depth).
4. Corrects the ambient socketlib type to the real shape.
5. Updates all tests to the real shape, preserving the 17-handler invariant.
6. Confirms the two DM-visible world settings persist + display.
7. Bumps the module to 0.1.7 + adds a changeset.

Purpose: restore the only missing link — the module actually emitting `/internal/delta` pushes — so the
already-deployed bridge receives the roster/bearer and real Forge pairing works.

Output: a foundry-module 0.1.7 whose `ready` hook never crashes and whose push readers always register.
</objective>
