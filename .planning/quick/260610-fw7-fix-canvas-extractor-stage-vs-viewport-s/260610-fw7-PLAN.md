<objective>
Fix horizontal-stripe frame corruption observed on the REAL Forge client (live evidence 2026-06-10, scene N6y1vDsWHaBYp5ms). `extractCurrentFrame` calls `renderer.extract.pixels(canvas.stage)` — PIXI v7 (Foundry v13/v14) renders a display-object target to a texture sized by the OBJECT'S LOCAL BOUNDS × resolution (the whole world, e.g. 4000×3000), but the code interprets the returned buffer with `renderer.width/height` → row-stride mismatch → striped garbage on the glasses. Synthetic tests never caught it because fixtures always returned a renderer-sized buffer.

This also lands the Phase 27 "viewport extraction" decision: capturing the viewport (what the player actually sees, with zoom + fog applied) instead of re-rendering the whole world stage offscreen every ~1 Hz capture.

Purpose: Stop emitting corrupted frames; show the player the live viewport; cut per-capture cost.
Output: Updated `canvas-extractor.ts` (viewport capture + fail-loud sanity guard), updated tests, patch changeset.
</objective>

    - After read, expected = srcWidth*srcHeight*4 where srcWidth/srcHeight = renderer.width/height.
    - If srcBytes.length === expected: proceed with renderer dims (the common viewport case).
    - If mismatch: compute k = sqrt(srcBytes.length / expected). If k is a clean integer >= 2 within float epsilon (e.g. abs(k - round(k)) < 1e-6 and round(k) >= 2 and (w*k')*(h*k')*4 === srcBytes.length), reinterpret dims as srcWidth*k' x srcHeight*k' and proceed; the box-average fit-downscale handles any source size natively.
    - If mismatch and no clean integer k>=2: console.warn with BOTH the expected and actual byte lengths and return null (skip frame).
    - The downscale loop and normalize pass MUST index using the (possibly inferred) effective dims, not the raw renderer dims.
  </behavior>
  <action>
    In `extractCurrentFrame`: change the extract call from `renderer.extract.pixels(canvas.stage)` to `renderer.extract.pixels()` (no target — PIXI v7 Extract reads the main framebuffer at renderer.screen x resolution, which equals renderer.width x renderer.height). Keep the existing try/catch around the call unchanged.

    After a successful read, introduce effective source dims (e.g. `effWidth`/`effHeight`) initialized to `srcWidth`/`srcHeight`. Compute `expected = srcWidth * srcHeight * 4`. If `srcBytes.length === expected`, use renderer dims as-is. Else compute `k = Math.sqrt(srcBytes.length / expected)`; let `kRound = Math.round(k)`; if `kRound >= 2` and `Math.abs(k - kRound) < 1e-6` and `(srcWidth*kRound)*(srcHeight*kRound)*4 === srcBytes.length`, set `effWidth = srcWidth*kRound`, `effHeight = srcHeight*kRound`. Otherwise `console.warn('[EVF canvas-extractor] pixel buffer length mismatch — expected', expected, 'got', srcBytes.length, '; skipping frame')` and `return null`.

    Replace EVERY subsequent use of `srcWidth`/`srcHeight` in the scale computation, the box-average downscale loop (the `sy*srcWidth` stride and the `srcWidth`/`srcHeight` clamps), with the effective dims. The cleanest approach: after inference, reassign — but `srcWidth`/`srcHeight` are `const`; introduce mutable `effWidth`/`effHeight` (let) and use those from the scale line (line ~234) downward, OR rename so the whole downscale region reads the effective dims. Do NOT leave any read of `renderer.width/height` feeding the stride math after inference (INV-4: no stale-dim bug path).
