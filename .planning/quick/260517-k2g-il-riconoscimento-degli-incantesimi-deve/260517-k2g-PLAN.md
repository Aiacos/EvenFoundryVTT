# Quick Task 260517-k2g: Entity Recognition — Foundry-Derived (Items + Actors)

> "il riconoscimento degli incantesimi deve valere per qualsiasi cosa, oggetti, armi, armature, PNG, Mostri ecc"

Estendere il riconoscimento vocale derivato-da-Foundry — già live per gli **spell** via `spell-lookup-foundry-derived` (chiuso 71 min fa) — a **qualsiasi entità giocabile**: armi, armature, oggetti, consumabili, tool, loot, contenitori, feat, PNG e mostri (Actor type=`npc`), vehicle.

## Architectural decision (LOCKED)

**ADDITIVE PARALLEL PIPELINE.** NON si rifattorizza spell-pack: si aggiunge un secondo pipeline `entity-pack` che copre tutto ciò che spell-pack NON copre. Spell-pack rimane immutato (53 test passanti, appena shipped). La duplicazione del codice è accettata; la deduplicazione futura è OUT OF SCOPE.

```
[Foundry game.packs] ──┐ Item (non-spell) + Actor (npc/vehicle)
                       │ Hooks: init / updateCompendium (debounce 500ms)
                       ▼
[foundry-module: entity-pack-reader.ts]
                       │ readAvailableEntities() emette envelope
                       ▼
[r1.entities.available envelope] ──→ /internal/delta onDelta hook
                                          │
                                          ▼
                              [bridge: EntityPackCache singleton]
                                          │ REST GET /v1/entities/available (bearer auth)
                                          ▼
                              [foundry-mcp: entity-lookup-foundry.ts]
                                  ↑ fetchAvailableEntities() + 5-min TTL
                                  └── fuzzy Levenshtein on dynamic list
                                  └── NO static fallback (returns null)
```

**socketlib invariant preserved.** Nessun nuovo `registerComplexHandler`. Conta = **17** (Phase 13 invariant) verificata via grep filtrato `.d.ts`.

## Scope

### Item subtypes inclusi (filter `pack.metadata.type === 'Item'`, `entry.type ∈`):
- `weapon` — armi
- `equipment` — armature + indossabili (scudi inclusi)
- `consumable` — pozioni, pergamene, munizioni
- `tool` — strumenti
- `loot` — tesori, gemme
- `container` — contenitori (backpack, pouch)
- `feat` — feature di classe / razza (es. "Action Surge")

### Actor subtypes inclusi (filter `pack.metadata.type === 'Actor'`, `entry.type ∈`):
- `npc` — PNG + mostri (dnd5e usa lo stesso discriminator per entrambi)
- `vehicle` — veicoli

### Esclusi (per scelta esplicita):
- Item.spell — già coperto da `spell-pack` (non duplicare)
- Item.class, subclass, background, race — non voice-actionable
- Actor.character — entità uniche del world, non template-target
- Actor.group — non voice-actionable

## Schema design (shared-protocol)

```
EntityPackEntrySchema {
  id: string.min(1)               // compendium _id, dedup key
  packId: string.min(1)           // 'dnd5e.items', 'dnd5e.monsters', etc.
  entityKind: 'item' | 'actor'    // discriminator a livello di pack (metadata.type)
  entityType: string.min(1)       // sub-type dnd5e: 'weapon' | 'equipment' | 'npc' | 'vehicle' | …
  name: string.min(1)             // canonical EN
  nameLocalized: string.min(1)    // game.i18n.localize(entry.name)
}

AvailableEntitiesPayloadSchema {
  entries: EntityPackEntrySchema[]
  source: 'foundry-packs' | 'empty'   // identici sentinel a spell-pack
  count: number.int.min(0)
  generatedAt: number.int.min(0)
}

R1_ENTITIES_AVAILABLE_TYPE = 'r1.entities.available' as const
```

Note: niente `level`/`school` (spell-only); il consumer voice-resolver usa solo `name`/`nameLocalized` per fuzzy match. `entityKind` + `entityType` sono ritornati dal lookup MCP per consentire al caller di dispatchare alla giusta azione Foundry (weapon-attack vs use-item vs target-token, ecc.) — questo plan NON wire il dispatch downstream; espone solo `kind/type/id/packId/name` al chiamante.

## Threat model

- **T-EP-01 (Injection)**: `name` e `nameLocalized` sono stringhe verbatim dal compendio Foundry. Nessun eval, shell, template injection. Stesso pattern T-SP-01 di spell-pack.
- **T-EP-02 (Cache poisoning)**: `AvailableEntitiesPayloadSchema.safeParse` PRIMA di ogni cache write (bridge handler) E PRIMA di costruire la lookup map (foundry-mcp). Stesso pattern T-SP-02. Mitigato con test esplicito (entities.test.ts caso "invalid payload → cache NON scritta" + entity-lookup-foundry.test.ts caso "invalid payload → null").
- **T-EP-03 (Stale cache after pack uninstall/update)**: hook `updateCompendium` re-emit, debounced 500ms. Stesso pattern T-SP-03.
- **T-EP-04 (Memory blowup su compendi grandi)**: bestiari D&D 5e completi possono superare diverse migliaia di entries (Monster Manual ~500, sommati a homebrew/expansion). Schema permette payload ampi (nessun cap nello schema). Mitigation: `console.warn` quando `entries.length > 10000` nel reader (vedi Task 1 action). Nessun hard cap richiesto per MVP — solo telemetria visibile in Foundry console per DM.
