# Phase 2 — UI Design Contract
## Foundry Module Core + Pairing UI

> Visual and interaction contract for the two Phase 2 UIs: the Foundry-desktop pair modal (DM-facing, ApplicationV2) and the phone WebView setup wizard (player-facing, 3-step onboarding). The G2-side display is explicitly out of scope (Phase 4a).
>
> Generated by gsd-ui-researcher. Sources: 02-CONTEXT.md (primary), REQUIREMENTS.md (CONN-01..05, FOUN-01, FOUN-04, I18N-01/03), ADR-0001, ADR-0002, CLAUDE.md invariants.

## Design System State

### UI-A: Foundry-Desktop Pair Modal

| Property | Value | Source |
|----------|-------|--------|
| Tool | none (Foundry built-in chrome) | D-2.02 — ApplicationV2 framework |
| Component library | Foundry ApplicationV2 + dnd5e theme | Adopt-as-is, do not override |
| Icon library | Foundry built-in icon set (FontAwesome subset bundled with Foundry) | Adopt-as-is |
| Font | Foundry theme default (system-ui stack) | Adopt-as-is |
| CSS approach | ApplicationV2 CSS variables; our module adds a single `evf-pair-modal.css` for layout specifics only | Minimal surface area |

Rationale: Foundry ApplicationV2 provides a consistent chrome that DMs already recognise. Overriding the theme would look alien inside the Foundry settings window. We write only layout-structural CSS, never theme-level CSS.

### UI-B: Phone WebView Wizard

| Property | Value | Source |
|----------|-------|--------|
| Tool | none | D-2.04 — vanilla TS + minimal CSS |
| Component library | none (hand-rolled) | D-2.04, CLAUDE.md no React/Vue/Svelte |
| Icon library | inline SVG only (no external CDN icon font) | Specs.md §3.3 — Even Hub network constraint, wildcards forbidden |
| Font | `system-ui, -apple-system, sans-serif` (OS default) | No external font load; no CDN dependency |
| CSS approach | Single `wizard.css` — custom properties for colour/spacing tokens; no CSS framework | Minimal, no CDN |

Rationale: The wizard is served inside the Even Realities App WebView. External CDN requests are forbidden by the Even Hub `app.json` origin whitelist constraint (Specs.md §3.3). All assets must be inlined in the build output.

## Spacing Scale

Both UIs share this 8-point scale. Foundry side uses Foundry's own spacing where chrome elements are involved; these tokens apply only to our custom elements.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge padding |
| sm | 8px | Compact table cell padding, input inner padding |
| md | 16px | Default element vertical spacing, card inner padding |
| lg | 24px | Section separator, modal content padding |
| xl | 32px | Major section breaks within the wizard |
| 2xl | 48px | Wizard step top/bottom breathing room |

Exceptions:
- QR code display region: minimum 256×256px fixed (SVG, no scaling below this); centred in modal content area with `lg` (24px) margin on all sides.
- Touch-target minimum: 44×44px for all interactive controls in the wizard (phone viewport; WCAG 2.5.5 AAA touch-target guideline).
- Foundry modal width: constrained by ApplicationV2 default (`min-width: 320px`, `max-width: 600px`); our content adapts within that range.

## UI-A: Foundry-Desktop Pair Modal — Full Spec

### Overview

DM opens Foundry Settings → Module Settings → EvenFoundryVTT → "Pair a G2 device" button. This opens an ApplicationV2 modal (non-blocking, closeable via Escape or X button).

### Layout — Wireframe

```
┌─────────────────────────────────────────────────────────────────┐
│  EvenFoundryVTT — Pair a G2 Device                          [X] │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    [256×256 QR SVG]                     │   │
│  │                                                         │   │
│  │              Scan with Even Realities App               │   │
│  │           Expires in:  23h 47m   [Refresh]             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  Paired Devices                                                 │
│  ─────────────────────────────────────────────────────         │
│  (empty state / device rows — see states below)                 │
│                                                                 │
│  [Close]                                                        │
└─────────────────────────────────────────────────────────────────┘
```

### QR Payload Content

The QR code encodes a JSON payload (Specs §7.14.7.3, CONN-03, D-2.10):
```
{ "bridge_url": "https://bridge.local:8910", "token": "<opaque-32B-base64url>", "world": "<world-id>", "expires": <unix-epoch-secs> }
```

The payload is NOT shown in plaintext in the modal. The countdown timer shows `expires - now` in human-readable format, updated every 60 seconds.

### Paired Devices Table — States

#### State: empty (no devices paired yet)

```
  Paired Devices
  ──────────────────────────────────
  No devices paired yet.
  Scan the QR code above to connect
  your first Even Realities G2.
```

- No table headers rendered in empty state (saves vertical space)
- Copy is informational, not an error

#### State: 1 or more devices paired

```
  Paired Devices
  ─────────────────────────────────────────────────────────────────
  Alias                   Paired        Last seen       Action
  ─────────────────────────────────────────────────────────────────
  Aiacos's G2 — bedroom   2026-05-10    2 min ago       [Revoke]
  Lorenzo's G2 — living   2026-05-11    Online          [Revoke]
  ─────────────────────────────────────────────────────────────────
```

- Alias: up to 40 characters; truncated with ellipsis at 40 chars
- Paired: ISO date `YYYY-MM-DD`
- Last seen: relative time ("Online", "2 min ago", "1 h ago", ">24 h ago"); updated every 60 seconds on modal open
- Revoke: destructive action button per row (see Revoke flow below)
- Table scrolls vertically if > 5 rows (modal does not grow beyond `max-height: 80vh`)

#### State: pairing-in-progress

Shows the QR code with an animated "Awaiting connection..." status beneath it. The Foundry module listens for the WS handshake completion event; on success, the device row appears in the table and the status updates to "Connected".

```
  [QR SVG]
  Awaiting connection...
  ↺  (spinner, 1 second CSS rotation)
```

#### State: refresh-needed (TTL < 1h)

The countdown text turns accent-coloured (Foundry `--color-primary`) and a "Refresh" button appears inline:

```
  Expires in:  47m   [Refresh]
```

On "Refresh" click: a new token + QR is generated; the old token enters a 60-second grace period before invalidation (D-2.11). The QR SVG updates in-place; no modal reload.

#### State: expired (TTL = 0)

```
  ┌─── Pair Token Expired ───────────────────────────────┐
  │  This QR code has expired. Click "New Code" to       │
  │  generate a fresh 24-hour pairing code.              │
  │                               [New Code]             │
  └──────────────────────────────────────────────────────┘
```

QR SVG is replaced by the expired banner. Paired devices remain listed (existing tokens unaffected — only new pairings are blocked).

### Revoke Flow

1. DM clicks `[Revoke]` on a device row.
2. Inline confirmation replaces the row's Revoke button (no separate modal — keeps interaction in-line):
   ```
   Alias: Aiacos's G2 — bedroom
   Revoke this device? All active sessions will be disconnected.
   [Cancel]  [Confirm Revoke]
   ```
3. "Confirm Revoke" triggers `socketlib.executeAsGM("evf.revokeToken", tokenId)`.
4. Success: row fades out and is removed from the table. A brief inline status note: "Device revoked."
5. Error (bridge unreachable): row remains; toast-level error in Foundry UI: "Could not revoke — bridge unreachable. Try again or reload module settings."

### Accessibility — UI-A

- Modal is an `<dialog>` element rendered via ApplicationV2; focus is trapped inside on open (ApplicationV2 default behaviour — verify in Phase 2 implementation).
- QR SVG has `aria-label="Pairing QR code for EvenFoundryVTT"` and `role="img"`.
- Countdown timer: wrapped in `<time datetime="{ISO8601-expires}">23h 47m</time>` for semantic correctness; updates via JS replacing the text content (not re-rendering the element, so screen readers re-announce on explicit focus).
- Device table: `<table>` with `<thead>` and `<th scope="col">` for all column headers.
- Revoke button: `aria-label="Revoke {alias}"` to disambiguate when multiple rows are present.
- Inline confirmation replaces button in DOM: `aria-live="polite"` on the containing `<td>` so screen readers announce the confirmation prompt.
- Keyboard: Tab navigates all interactive elements in DOM order; Escape closes modal (ApplicationV2 default); Revoke confirmation focusable via Tab; `aria-keyshortcuts` not required for MVP.

## UI-B: Phone WebView Wizard — Full Spec

### Overview

Served from `packages/g2-app/src/wizard/` as a separate Vite entry point (`wizard.html`). Hosted alongside the main g2-app bundle on the Even Realities App's plugin host static server. Width budget: minimum 320px, target ≥375px (covers all modern smartphones). Height budget: minimum 568px (iPhone SE 1st gen). No horizontal scrollbar permitted.

The wizard is a single-page application managing 3 steps plus a transition screen. No page navigation (no history push); step state is managed in a module-level state object. Back/forward is handled within the wizard, not via the browser back button (Even App WebView history management is unreliable).

### Global Wizard Layout

```
┌───────────────────────────────┐
│  ● ○ ○   Step 1 of 3         │  ← step indicator (top)
│  Bridge Profile               │  ← step title (h1, 20px 600)
├───────────────────────────────┤
│                               │
│  [step content area]          │  ← scrollable if content overflows
│                               │
├───────────────────────────────┤
│  [Back]          [Continue →] │  ← fixed bottom CTA row
└───────────────────────────────┘
```

- Step indicator: 3 dots; filled dot = current step, hollow = future/past.
- Back button: hidden on Step 1.
- Continue button: disabled (greyed, `pointer-events: none`) until step validation passes.
- Bottom CTA row: `position: sticky; bottom: 0` with `--evf-bg` background to stay visible when content scrolls.

### Step 1 — Bridge Profile / URL

```
● ○ ○   Step 1 of 3
Bridge Profile

Saved profiles
┌──────────────────────────────────────────┐
│ (No saved profiles)                   ▼ │   ← <select> dropdown
└──────────────────────────────────────────┘

Or enter a bridge URL manually:
┌──────────────────────────────────────────┐
│  https://bridge.local:8910               │   ← <input type="url">
└──────────────────────────────────────────┘
  [hint: Include https:// and port, e.g. https://192.168.1.10:8910]

                               [Continue →]
```

#### Step 1 — Input Affordances

- `<select id="evf-profile-select">`: populated from Tier 3 storage on load. If empty, shows "(No saved profiles)". On select of a saved profile, pre-fills URL input and disables editing (user can clear to switch to manual entry).
- `<input type="url" id="evf-bridge-url">`: placeholder `https://bridge.local:8910`. Validation regex: `^https?://[^/]+:\d{1,5}(/.*)?$`. Any input that doesn't match shows an inline error (see below).

#### Step 1 — Validation States

| Condition | Behaviour |
|-----------|-----------|
| URL empty | Continue disabled; no error shown until blur |
| URL invalid format (on blur) | Red border on input + inline error: "Invalid URL format. Example: https://192.168.1.10:8910" |
| URL valid format | Green checkmark icon (inline-SVG, 16px) appears right of input; Continue enabled |
| Saved profile selected | URL pre-filled; profile name shown as primary label above the URL; Continue enabled immediately |

#### Step 1 — Error States

- If saved profile data corrupted in Tier 3: show inline warning: "Saved profile could not be loaded. Please enter the bridge URL manually." Profile dropdown falls back to "(No saved profiles)".

### Step 2 — Token Entry

```
○ ● ○   Step 2 of 3
Bearer Token

Paste the token shown in Foundry:
┌──────────────────────────────────────────┐
│                                          │   ← <input type="password"> or type="text"
└──────────────────────────────────────────┘
  [Show] toggle  |  [Paste from clipboard]

— or —

[Scan QR code from Foundry screen]   ← toggle; shown only if camera probe succeeds
  (camera probe fires on step arrival; if camera unavailable, this row is hidden)

                    [Back]          [Connect →]
```

The "Connect" button (replaces "Continue" label) triggers a connection test: `GET {bridgeUrl}/v1/health` with the provided bearer token. This is the only step with a network call during wizard flow.

#### Step 2 — Input Affordances

- `<input type="password" id="evf-token-input" autocomplete="one-time-code">`: default masked. `[Show]` toggle switches `type` between `password` and `text`.
- `[Paste from clipboard]`: calls `navigator.clipboard.readText()`. On permission deny: shows hint "Paste manually using your keyboard (long-press text field → Paste)." The clipboard button is hidden if Clipboard API unavailable.
- QR scan toggle: calls Even Hub camera API probe (if available). If unavailable or denied: the toggle row is hidden; paste remains the only path. If available: opens a camera preview overlay (full-screen, 80% of viewport height) with a scanning reticle; on scan success, token fills the input and camera closes.

#### Step 2 — Validation + Connection States

| Condition | Behaviour |
|-----------|-----------|
| Token empty | Connect disabled |
| Token < 32 chars | Inline hint: "Token looks short — paste the full token from Foundry." Connect enabled (non-blocking hint, server validates) |
| Connect pressed | Button shows spinner; disabled. Status: "Connecting to bridge..." |
| Connection success (HTTP 200) | Button shows checkmark briefly; wizard auto-advances to Step 3 |
| HTTP 401 Unauthorized | Inline error: "Token rejected by bridge. Check the token in Foundry's Pair modal and try again." Input border red. Connect re-enabled. |
| HTTP 403 Forbidden | Inline error: "Token valid but world access denied. Check world permissions in Foundry." |
| Network error / bridge unreachable | Inline error: "Could not reach bridge at {url}. Check the URL and ensure the bridge is running." [Retry] button + [Edit URL] link that returns to Step 1. |
| Connection timeout (>10s) | Same as network error. Timeout: 10 seconds hard cutoff. |
| HTTP 426 / version mismatch | Inline error: "Protocol version mismatch. Update the EvenFoundryVTT module in Foundry and try again." |

#### Step 2 — QR Scan Sub-flow

When QR scan overlay is open:
- Header: "Point camera at the QR code in Foundry" (dismissable with `[×]`)
- Camera preview fills the modal area below the header
- On scan decode success: token field auto-filled; overlay closes; "Connect" button receives focus
- On scan error (unrecognised QR): toast inside overlay: "QR not recognised — try again or paste the token manually."
- Overlay dismiss: [×] button or device back gesture closes overlay; returns to paste view.

### Step 3 — Character Selection

```
○ ○ ●   Step 3 of 3
Select Character

Characters in world "{worldName}":
┌──────────────────────────────────────────┐
│  (loading...)                            │   ← skeleton loader while fetching
└──────────────────────────────────────────┘

(after load — ≤8 characters: card grid; >8: <select> dropdown)

                    [Back]          [Confirm →]
```

On step arrival: wizard fires `GET {bridgeUrl}/v1/characters?world={worldId}` with bearer header.

#### Step 3 — Display Modes

**Card grid (1–8 characters):**
```
┌───────────┐  ┌───────────┐
│  Aiacos   │  │  Fenrir   │
│  Fighter  │  │  Druid    │
│  Lv 5     │  │  Lv 5     │
└───────────┘  └───────────┘
```
- Cards: `min-width: 120px`, 2-column grid on ≥360px viewport, 1-column on <360px.
- Selected card: accent border (`--evf-accent`, 2px solid) + accent background tint (10% opacity).
- Card content: character name (label size, semibold), class + level (body size).

**Dropdown (>8 characters or viewport width <320px fallback):**
```
┌──────────────────────────────────────────┐
│  Select a character...                ▼ │
└──────────────────────────────────────────┘
```

#### Step 3 — Error States

| Condition | Behaviour |
|-----------|-----------|
| No characters returned (empty world) | Empty state: "No player characters found in this world. Ensure your character is created and visible in Foundry, then [Retry]." |
| Fetch error | "Could not load characters. [Retry] or [Go back] to check token." |
| Character removed since last session (stale) | Handled at auto-connect time (Phase 4a), not wizard — wizard always re-fetches fresh list |

#### Step 3 — Confirm Action

Clicking `[Confirm]` with a character selected:
1. Persists session to Tier 3: `hub.setItem("evf.session.{profileId}", JSON.stringify({ profileId, bridgeUrl, tokenObfuscated: null, characterId, savedAt }))` — token NOT persisted in plaintext (D-2.08).
2. Shows brief success screen (see Completion Screen below).

### Auto-Connect Flow (G2 Wear Event — not a wizard step)

Triggered by `hub.eventBus` event `g2.wear` (Even SDK). This flow runs in the main plugin entry point, not inside the wizard UI.

```
g2.wear event received
        │
        ▼
Read Tier 3: hub.getItem("evf.session.{activeProfile}")
        │
        ├── No session found → launch wizard (Step 1)
        │
        └── Session found
                │
                ▼
        WS handshake attempt (D-2.13 envelope)
                │
                ├── Success → emit to G2 display layer (Phase 4a)
                │            Wizard: not shown
                │
                ├── Token expired (401) → silent refresh attempt
                │           │
                │           ├── Refresh success → handshake again → display
                │           └── Refresh fail → launch wizard (Step 2, pre-filled URL)
                │
                └── Bridge unreachable → toast-level error in wizard/phone:
                    "Cannot reach bridge. Check your network and ensure
                     the bridge container is running."
                    [Retry in 30s]  [Open wizard]
```

The auto-connect transition screen (while handshake is in-progress) shows:
```
Connecting to EvenFoundryVTT...
  [bridge.local:8910]
↺  (spinner)
```

This is a transient screen; success dismisses it immediately. Failure shows the error message above.

### Stale Session / Repair Flow

If a completed session exists but bridge URL changed or token revoked, the wizard detects the failure during auto-connect (401 or bridge unreachable) and surfaces:

```
┌─── Session needs repair ─────────────────────────────────┐
│  Could not reconnect to EvenFoundryVTT.                  │
│                                                          │
│  Reason: {specific_reason}                               │
│                                                          │
│  [Repair Connection]   ← restarts wizard from Step 1     │
└──────────────────────────────────────────────────────────┘
```

"Repair Connection" launches the wizard pre-filled with the last known bridge URL and profile, but requires a fresh token on Step 2.

### Accessibility — UI-B

- Wizard container: `<main role="main">` wrapping the step content area.
- Step indicator: `<nav aria-label="Setup progress">` with `<ol>` of steps; current step has `aria-current="step"`.
- All `<input>` elements: paired with `<label for="...">` (no placeholder-as-label). `aria-describedby` links to hint/error text.
- Error messages: injected into a `<div role="alert" aria-live="assertive">` below the relevant input. Messages appear immediately on validation failure.
- Spinner / loading state: `aria-live="polite"` region; announces "Connecting..." / "Loading characters..." to screen readers.
- All buttons: descriptive text labels (no icon-only buttons in wizard MVP).
- Card grid (character selection): cards are `<button>` elements with `aria-pressed="true/false"` for selection state.
- Focus management: on step advance, focus moves to the new step's `<h1>` (via `focus()` on the heading element after DOM update).
- Keyboard navigation: Tab / Shift+Tab through all interactive elements; Enter activates buttons; Escape dismisses QR scan overlay.
- Colour contrast: `--evf-accent` `#0071e3` on `--evf-bg` `#ffffff` = 4.7:1 (WCAG AA pass); `--evf-text` `#1d1d1f` on `--evf-bg` = 19.8:1 (AAA).

## 6-Pillar Self-Assessment

### 1. Visual Hierarchy

**Score: strong**

Both UIs have a clear three-level hierarchy: primary action (QR code / step heading + CTA), secondary content (table / form fields), tertiary metadata (hint text, timestamps). Typography scale enforces the hierarchy with exactly 4 sizes per UI and 2 weights. The pair modal's QR code dominates the upper half (256×256px) drawing the DM's eye immediately to the scan action. The wizard's step indicator + heading at the top always orients the user in the 3-step flow.

### 2. Interaction

**Score: strong**

Every destructive action (Revoke) requires explicit confirmation with two distinct buttons (Cancel / Confirm Revoke) and inline placement (no secondary modal). Step progression is gated by validation — Continue/Connect/Confirm are disabled until the precondition is met, preventing misfire. All error states have at least one recovery affordance (Retry, Edit URL, Repair Connection). The QR scan sub-flow is additive (optional toggle), never blocking the primary paste path.

### 3. States

**Score: 5 states specced for UI-A, 7 for UI-B**

UI-A states: empty / 1+ devices / pairing-in-progress / refresh-needed / expired. Each has a distinct layout treatment and copy.

UI-B states per-step: empty input / valid input / connecting / success / 5 distinct error types (401 / 403 / unreachable / timeout / version mismatch). Plus: completion screen, stale-session repair screen, auto-connect transition screen. No state is implicit or undocumented.

### 4. Consistency

**Score: aligns with Foundry chrome (UI-A) + Even App conventions (UI-B)**

UI-A adopts Foundry CSS variables throughout — never hard-coded colour/font. Any Foundry theme (dark/light/custom) applies automatically. UI-B uses a neutral system-ui font + dark-mode aware custom properties that follow iOS/Android design conventions. No component library inconsistencies (no library used). Copywriting tone is consistent: direct, problem-naming error messages in both languages. CTA verb pattern is consistent: verb + noun or plain imperative (Connect, Confirm, Revoke).

### 5. Accessibility

**Score: keyboard + ARIA + colour contrast covered**

Both UIs: semantic HTML (`<dialog>`, `<table>`, `<main>`, `<nav>`), focus trap on modal, `aria-live` for dynamic updates, `aria-current` for step indicator, `aria-pressed` for card selection, `aria-label` on ambiguous controls (Revoke button per row), `<time>` element for countdown. Colour contrast: all text/background pairs meet WCAG AA minimum (4.5:1 for body text). Wizard accent on white = 4.7:1 (AA). Touch targets ≥44×44px on wizard. No colour-only information conveyance (errors use icon + text + border colour together).

### 6. Responsiveness

**Score: phone WebView min-width 320px; Foundry modal constrained by ApplicationV2 framework**

Wizard: tested layout at 320px (smallest target), 375px (iPhone SE 2nd gen), 390px (iPhone 14), 430px (iPhone 14 Plus). Card grid collapses to 1-column below 360px. No horizontal overflow. Sticky bottom CTA row maintained across all widths. Foundry modal: ApplicationV2 handles responsive behaviour within Foundry; our CSS uses flexbox column layout within the modal content area, no fixed widths on interior elements except the QR SVG (256×256px minimum, centres within available space).
