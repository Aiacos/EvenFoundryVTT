<threat_model>
T-02-01 [HIGH]: Bearer token leak — token is NEVER persisted to Tier 3 in plaintext (D-2.08). `tier3-storage.ts` schema explicitly excludes any `token` field. Token held in module-level memory only during the wizard session; discarded on wizard close. `tokenObfuscated` field in storage is always `null` in Phase 2.
T-02-03 [MEDIUM]: Wizard XSS — all user-supplied content (alias, bridgeUrl, characterName) rendered via `element.textContent = ...` (never `.innerHTML`). The only innerHTML injection is the i18n string for the hint text (loaded from trusted bridge endpoint, not user input). Validate i18n response is a plain object with string values before rendering.
T-02-04 [MEDIUM]: Even Hub kv store poisoning — `loadSession` validates loaded JSON against a Zod schema before using any field. If schema validation fails, session is treated as corrupted and wizard starts fresh with an informational message (D-2.08 corrupt profile case).
</threat_model>

## Task Breakdown

### Task 1: Wizard infrastructure — state machine, Tier 3, i18n

**Files:**
- `packages/g2-app/src/types/even-hub.d.ts` — expand with `hub.camera?`, `hub.eventBus` types
- `packages/g2-app/src/wizard/state.ts` — wizard state machine: `WizardStep` enum (STEP1 | STEP2 | STEP3 | COMPLETION | REPAIR), `WizardState` interface (`{ step, bridgeUrl, token, characterId, profileId, i18n, error }`), `createStore()` function returning a reactive state store (simple `{ get, set, subscribe }` pattern — no external library)
- `packages/g2-app/src/wizard/tier3-storage.ts` — `saveSession(s: Session): Promise<void>` calls `hub.setItem("evf.session.{s.profileId}", JSON.stringify(s))`; `loadSession(profileId: string): Promise<Session | null>` calls `hub.getItem`, parses JSON, validates with `SessionSchema.safeParse`, returns null on failure with console.warn (biome-ignore line); `listProfiles(): Promise<Session[]>` returns all stored sessions
- `packages/g2-app/src/wizard/i18n.ts` — `loadI18n(bridgeUrl: string, lang: string): Promise<Record<string, string>>` fetches `GET {bridgeUrl}/v1/i18n/{lang}` with 5s timeout, validates response is `Record<string, string>`, caches in module-level Map; `t(key: string): string` returns cached string or key as fallback (never throws)
- `packages/g2-app/vite.config.ts` — add `wizard: "src/wizard/wizard.html"` to rollupOptions.input
- `packages/g2-app/src/wizard/wizard.test.ts` — tests for tier3-storage (mock hub), i18n loader (mock fetch), state machine transitions

**Action:**
The state machine is a hand-rolled observable store: `createStore<T>(initial: T)` returns `{ get(): T, set(partial: Partial<T>): void, subscribe(fn: (s: T) => void): () => void }`. No external state library. The `set` merges partial updates via spread and calls all subscribers. This is the only state coordination mechanism — step components import and call `store.set({ step: WizardStep.STEP2 })`.

`tier3-storage.ts` uses Zod for schema validation (`@evf/shared-protocol` is not yet populated in Wave 1 — define `SessionSchema` locally here, move to shared-protocol in Plan 05 refactor if needed). `tokenObfuscated: z.null()` enforces Phase 2 invariant at the type level.

`i18n.ts`: Locale detection from `navigator.language.split('-')[0]` (Even App browser locale), falling back to `'en'`. The `/v1/i18n/{lang}` endpoint does not exist yet (Plan 04 creates it) — i18n loader must handle `fetch` failure gracefully: log warn + return empty object (UI falls back to key names, which is acceptable for development).

### Task 2: Step components + wizard entry + auto-connect

**Files:**
- `packages/g2-app/src/wizard/wizard.html` — base HTML page with `<main role="main">`, step indicator `<nav aria-label="Setup progress">`, step content `<div id="step-content">`, bottom CTA row `<div class="wizard-cta">`. References `wizard.ts` via `<script type="module" src="./wizard.ts">`.
- `packages/g2-app/src/wizard/wizard.css` — custom properties `--evf-bg`, `--evf-surface`, `--evf-accent`, `--evf-text`, etc. from 02-UI-SPEC.md; prefers-color-scheme dark block; sticky bottom CTA; step indicator dots; touch-target min 44×44px; card grid with 2-col on ≥360px / 1-col below.
- `packages/g2-app/src/wizard/steps/step1-profile.ts` — renders Step 1 into `#step-content`. Profile `<select>` populated from `listProfiles()`. Bridge URL `<input type="url">`. Validation regex `^https?://[^/]+:\d{1,5}(/.*)?$` on blur. Enables Continue on valid URL. Calls `store.set({ bridgeUrl, step: WizardStep.STEP2 })` on Continue.
- `packages/g2-app/src/wizard/steps/step2-token.ts` — renders Step 2. Password input, Show toggle, Paste from clipboard button, optional QR scan toggle (probes `hub.camera?.requestAccess()`; hides if unavailable). Connect button fires `GET {bridgeUrl}/v1/health` with `Authorization: Bearer {token}` header, 10s timeout. On 200: `store.set({ token, step: WizardStep.STEP3 })`. On each error code: sets `store.set({ error: { type: "401" | "403" | "unreachable" | "timeout" | "version_mismatch", url: bridgeUrl } })` and renders error message from i18n.
- `packages/g2-app/src/wizard/steps/step3-character.ts` — renders Step 3. On mount: `GET {bridgeUrl}/v1/characters?world={worldId}` with bearer. Renders card grid if ≤8, dropdown if >8. On Confirm: `saveSession(session)` → `store.set({ characterId, step: WizardStep.COMPLETION })`.
- `packages/g2-app/src/wizard/steps/completion.ts` — renders Completion screen (D-2.07). No navigation buttons. If `loadSession(profileId)` exists on wizard re-open, shows this screen with Repair link.
- `packages/g2-app/src/wizard/auto-connect.ts` — `initAutoConnect(): void` called from `wizard.ts` on load. Registers `hub.eventBus.on("g2.wear", handleWear)`. `handleWear`: loads session → if none, `store.set({ step: WizardStep.STEP1 })`; if session found, attempts WS handshake via `openHandshakeWebSocket(session)` (stub that logs `// TODO (ADR-0002): Phase 3 fills real WS connect`); on 401: `store.set({ step: WizardStep.STEP2, error: { type: "401" } })`.
- `packages/g2-app/src/wizard/wizard.ts` — entry point: `initWizard()` calls `loadI18n()`, `initAutoConnect()`, subscribes to store, renders correct step component on each state change by replacing `#step-content` innerHTML via `step.render(container)`.

**Action:**
All DOM updates use `element.textContent = ...` or `element.setAttribute(...)` for dynamic content (never `.innerHTML` for user data — T-02-03 mitigation). The step component interface: each step file exports `render(container: HTMLElement, store: Store, i18n: I18nFn): void` and `destroy(): void` (for cleanup). `wizard.ts` calls `currentStep?.destroy()` before switching steps.

The `GET /v1/health` endpoint in Plan 04 will validate the bearer. For now, Step 2's `connect()` function should handle any `response.status` that is not 200/201 gracefully.

`auto-connect.ts` WS handshake is explicitly a stub with a descriptive comment `// TODO (ADR-0002): Plan 04 wires real WS connect. This stub logs the session and returns.` — the TODO has the correct format per INV-4.
