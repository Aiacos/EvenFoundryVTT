From packages/foundry-module/src/write-path/handlers/cast-spell.ts (handler reference pattern — copy structure):
```typescript
export const castSpellHandler: ToolHandler<(typeof CastSpellInputSchema)['_input']> = {
  argsSchema: CastSpellInputSchema,
  async handle(args): Promise<ToolResult> {
    const actor = game.actors.get(args.actor_id);
    if (actor === undefined) return { success: false, error: 'actor_not_found' };
    const item = actor.items?.contents.find((i) => i.id === args.spell_id);
    if (item === undefined) return { success: false, error: 'item_not_found' };
    const activity = item.system.activities?.contents[0];
    if (activity === undefined) return { success: false, error: 'no_activity' };
    try {
      const result = await activity.use({ configure: false, ...slotOverride });
      return { success: true, data: { chatCardId: extractChatCardId(result) } };
    } catch (err) {
      if (isNoGmError(err)) return { success: false, error: 'no_gm_connected' };
      return { success: false, error: String(err) };
    }
  },
};
```

<inline_research>
**D-13-01 — dnd5e 5.3.3 Shield spell shape (resolve in handler):**
- Shield is a level-1 abjuration spell. Casting cost: 1 reaction.
- Activity location: `actor.items.contents` → find item with `system.identifier === 'shield'` or fallback `name === 'Shield'` / locale match `name === 'Scudo'`.
  Use a defensive resolver: prefer `system.identifier === 'shield'` (5e SRD id), then English/Italian name fallback. Surface `'spell_not_known'` if none match.
- Activity shape: `item.system.activities?.contents` — first activity is `'cast'` type.
- Reaction slot accounting: dnd5e 5.3.3 does NOT have a `consume: { reaction: true }` config knob on `activity.use()`. Reaction consumption is enforced server-side via the spell's `system.activation.type === 'reaction'` setting and the actor's combat tracker. The handler does NOT need to pass `consume.reaction`; the cast itself consumes the reaction by virtue of being a 1-reaction-cost spell. Combat-action-tracker (Plan 13-02) handles the visible widget update.
- Spell slot consumption: Shield is a level-1 spell — supply `spell.slot: 'spell1'` override (matches cast-spell handler's slot_level=1 path). For Plan 13-01, accept an optional `slot_level: 1` argument; default to 1. NO upcast support in MVP (D-13-01 — Shield does not benefit from upcasting).
- Implementation: `await activity.use({ configure: false, spell: { slot: 'spell1' } })`.
- Verify in test via mocked `game.actors.get(...).items.contents.find(...)` returning a Shield item shape; assert `activity.use` called with `{ configure: false, spell: { slot: 'spell1' } }`.

**D-13-02 — Counterspell ability check resolution flow:**
- Counterspell is a level-3 abjuration spell. Casting cost: 1 reaction.
- 2024 PHB: Counterspell triggers a CON save by the target caster (DC = caster's spell save DC). 2014 PHB: ability check contest. Both are resolved by the dnd5e Activity workflow when `activity.use()` fires — the system automatically prompts the target for the appropriate roll. The handler does NOT coordinate the contested check.
- Item resolution: identical pattern to Shield — `system.identifier === 'counterspell'` (en) / name match (it: 'Contromagia').
- Slot override: counterspell is level-3 (`spell.slot: 'spell3'`); upcast is permitted (3..9 slots, increases auto-fail level threshold in 2014). Accept `slot_level: 3..9` (default 3) per cast-spell schema convention.
- Implementation: `await activity.use({ configure: false, spell: { slot: \`spell${slot_level}\` } })`.
- The handler returns success after activity.use resolves; the actual save/check outcome lives in the dnd5e chat card. Targeting (which spell to counter) is the player's R1 selection in Plan 13-02's prompt — passed as `target_id` for audit/logging but NOT consumed by activity.use (the activity itself targets whatever Foundry is currently resolving).

**D-13-03 — Opportunity Attack chain via dnd5e API path:**
- Opportunity Attack is a built-in Reaction in 5e: when a hostile creature leaves the player's reach, the player may use their Reaction to make ONE melee weapon attack.
- dnd5e 5.3.3 implementation: NO dedicated `evf.opportunityAttack` machinery in the system. The reaction is a standard melee weapon attack with the "Opportunity Attack" flavor.
- Implementation strategy: REUSE the weapon-attack handler path. The handler:
  1. Resolves the actor's primary melee weapon (or accepts `item_id` argument explicitly — preferred for determinism).
  2. Locates the attack activity on the weapon (`activity.type === 'attack'`).
  3. Invokes `activity.use({ configure: false, consume: { action: false } })` — `consume.action: false` because OA does NOT consume the player's Action slot (it consumes the Reaction slot instead, which is tracked by combat-action-tracker via the audit log; the player's Action remains available).
  4. Sets `flags.dnd5e.opportunityAttack: true` on the resulting chat card via the `messageConfig` parameter so the chat-card text reads "Opportunity Attack" instead of a vanilla attack. Pattern: `activity.use({...}, { flags: { dnd5e: { opportunityAttack: true } } })` — verify with mocked activity.use that the messageConfig flag is forwarded.
- Target argument (`target_id`) is captured into the audit log for repudiation; activity.use does not consume it directly.

**T-13-04 — Reaction handler bypass mitigation:**
- All three handlers route through `dispatchTool` (existing pipeline in tool-registry.ts). Bearer validation, idempotency-bound caching, and audit log writing are already in place via dispatchTool. Plan 13-01 does NOT add any new auth surface.
- Combat-action-tracker (Plan 13-02) will read `flags.evf.audit.toolId in [cast-shield, cast-counterspell, opportunity-attack]` to increment `reactionsUsed` — no separate tracking path.
- CI Gate 8 (`activity.use(` absent from g2-app + bridge) remains GREEN: all activity.use calls are inside `packages/foundry-module/src/write-path/handlers/`.
