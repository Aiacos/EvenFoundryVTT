<inline_research>
**D-13-05 — actor.img Foundry field shape:**
- Foundry actor documents expose `actor.img: string` as the canonical portrait URL field. Verified via Foundry v13 actor data model — `img` is a top-level field on every Document subclass.
- Default value: `'icons/svg/mystery-man.svg'` (Foundry's stock placeholder). Plan 13-03 emits `portrait.url` whether or not the URL is the placeholder — Plan 13-04 decides whether to render (the placeholder will dither to a recognizable greyscale silhouette, which is acceptable MVP behavior).
- URL shapes seen in practice: `worlds/{world-name}/portraits/{filename}.webp` (relative to Foundry root), `https://example.com/cdn/portrait.png` (absolute external), `data:image/png;base64,...` (rare; Foundry uploader writes files to disk).
- character-reader filter: emit `portrait: { url: actor.img }` only when `actor.img` is a non-empty string. Leave undefined otherwise. NO filtering of placeholder URLs at the reader (bridge handles validation).

**T-13-02 — Portrait URL untrusted mitigation:**
- The bridge proxy MUST validate the URL before fetching to prevent SSRF (the bridge is the only network-facing component; if a compromised character-reader produced a malicious URL, the bridge would fetch it).
- Validation rules in priority order:
  1. Parse URL with `new URL(actor.img, foundryWorldOrigin)`. Throws on malformed input → 400.
  2. Reject if scheme is not `http:` or `https:`. Data URIs rejected (renderer cannot consume them without extra decode plumbing — defer to a future milestone).
  3. Reject if hostname is in a deny-list of cloud metadata endpoints: `169.254.169.254`, `metadata.google.internal`, `metadata.azure.com`, `localhost` (string match), `127.0.0.1`, `0.0.0.0`. AND if hostname resolves to a private IP range — for MVP the hostname-string deny-list is sufficient; DNS rebinding is a Phase 14+ concern.
  4. Same-origin enforcement: the resolved URL hostname MUST match the configured Foundry world hostname (`process.env.EVF_FOUNDRY_ORIGIN_HOST`). Relative paths (`worlds/...`) resolve against the Foundry origin automatically via `new URL(actor.img, foundryOrigin)`.
- Configuration knob: `allowedHosts: string[]` injected into registerPortraitRoute — defaults to `[process.env.EVF_FOUNDRY_ORIGIN_HOST ?? '']`. Reuse the bridge's existing CORS-configured Foundry origin.
- Tests T-13-02-01..05: rejected for data URI, malformed URL, deny-listed hostname, mismatched hostname; accepted for same-origin absolute + relative path.

**T-13-03 — Portrait cache poisoning mitigation:**
- Cache key = `SHA-256(resolved-absolute-URL)`. Same URL → same bytes → same key. No actorId in the key (admin re-uploading a portrait at the same URL would invalidate by re-emit, not by key change — acceptable for MVP).
- Actor ownership re-check on each cache hit: route reads request bearer → resolves player → asserts `actorId` is in player's owned actors before returning bytes (foundrySnapshotFn already enforces this; route reuses the pattern).
- TTL: 1 hour. LRU max 32 entries — for a single-tenant homelab with 4-6 players that's ample.
- Tests T-13-03-01..03: cache hit returns cached bytes; cache expiry re-fetches; actor ownership mismatch returns 403 even on cache hit (defensive).

**D-13-06 — Bridge dependencies for the dither pipeline:**
- The bridge package needs: `image-q` (Floyd-Steinberg dither + custom 16-step greyscale palette) and `upng-js` (4-bit indexed PNG encode). Both already used by `@evf/g2-app` (Phase 4a).
- For resize: prefer `sharp` (zero-build native, widely deployed in Node). If `@napi-rs/canvas` is already in the workspace, use that instead. If neither: install `sharp`. Decision logged in SUMMARY.
- Pipeline at execute time:
  1. fetch(url) → Buffer. Foundry world assets are typically world-public; bridge fetches WITHOUT cookie/bearer. 403/401 → return 502 to client.
  2. sharp(buffer).resize(100, 60, { fit: 'cover' }).raw().toBuffer() → RGBA buffer.
  3. image-q PointContainer.fromUint8Array → palette quantize → Floyd-Steinberg dither against 16-step greyscale palette.
  4. upng-js encode with depth=4 indexed-palette.
  5. Return `{ pngBytes: Uint8Array, urlHash: sha256(url) }`.

**D-13-07 — r1.portrait.ready envelope vs HTTP route — both ship:**
- The bridge exposes BOTH:
  - GET `/v1/portrait/:actorId` (HTTP REST) — Plan 13-04's CharacterSheetPanel Bio tab calls this on tab-open; returns `200 image/png` with bytes + `ETag: <urlHash>`. The g2-app handles fetch + decode.
  - On successful render (cache-miss path), the bridge ALSO emits `r1.portrait.ready` WS delta envelope with `{actorId, pngBase64, width, height, urlHash}` so the panel doesn't roundtrip on first open.
- For Plan 13-03 simplicity, the HTTP route is the AUTHORITATIVE path; the WS push is an OPTIMIZATION. If WS-push wiring is too invasive, the executor MAY ship HTTP-only and document the push-path as deferred (acceptable deviation; surface in SUMMARY).
