```typescript
export function attachReactionToastHandler(ws, toastQueue): () => void {
  const handler = (event: { data: unknown }) => {
    try {
      const envParse = EnvelopeSchema.safeParse(JSON.parse(String(event.data)));
      if (!envParse.success) return;
      if (envParse.data.type !== R1_REACTION_AVAILABLE_TYPE) return;
      const payloadParse = ReactionAvailablePayloadSchema.safeParse(envParse.data.payload);
      if (!payloadParse.success) return;
      toastQueue.enqueue(buildToastFromReaction(payloadParse.data, locale));
    } catch (err) { console.warn(...); }
  };
  ws.addEventListener('message', handler);
  return () => ws.removeEventListener('message', handler);
}
```

<inline_research>
**D-13-04 — 500ms debounce semantics:**
- Goal: avoid flicker-mounting a prompt over a flurry of reaction-eligible NPC actions in the same combat tick. If two `r1.reaction.available` envelopes arrive within 500ms, the latter REPLACES the former in the timer (`clearTimeout(prev); setTimeout(mount, 500)`).
- If the panel is already mounted when a new envelope arrives, the new envelope is IGNORED (the user must resolve the current prompt first; this matches the toast queue's FIFO+squash discipline established Phase 4b TOAST-01).
- 5s auto-timeout: after mount, if no [Y]/[N] within 5000ms, dispatcher tears down the panel via `layerManager.bundle([{ type:'destroy', z: ZIndex.Z2_OVERLAY }])`. This satisfies T-13-01 (reaction prompt spam mitigation) — the prompt cannot stay on the glasses forever.

**T-13-01 — reaction prompt spam mitigation:**
- Three guards: (a) 500ms debounce coalesces same-tick envelopes; (b) max-one-prompt-at-a-time (concurrent envelopes silently dropped); (c) 5s auto-timeout. Tests RPD-DEBOUNCE-01..03 + RPD-CONCURRENT-01 + RPD-TIMEOUT-01 cover all three.

**Panel layout (UI-SPEC pattern, derived from conc modal §3.5):**
Per CONTEXT.md §Area 1 the mockup is:
```
┌─[ REAZIONE: Shield disponibile ]──────────────────────────┐
│                                                            │
│  Goblin attacca te (15 → 17 ▲)                             │
│                                                            │
│  [Y] Lancia Shield (-1 reaz)                               │
│  [N] Annulla                                                │
│                                                            │
└────────────────────────────────────────────────────────────┘
```
- Title bracket includes the localized kind (Shield / Contromagia / Att. Opportunità).
- Subject row uses `sourceName` from the payload — the watcher's "actor.name" (truncate to 30 cp budget).
- Y/N rows are separate lines below the subject; double-tap=N is the same gesture convention as concentration modal.
- The `(15 → 17 ▲)` roll preview is REACT-01 toast info — for Shield only; for Counterspell/OA the subject row uses a different template (per i18n keys above). Plan 13-02 does NOT compute the roll preview; it just renders `payload.sourceName` plus the subject template. Roll preview enrichment is future work (post-milestone).

**Reactions slot accounting (D-13-05):**
- combat-action-tracker.ts already has `ActorEconomyState.reactionsUsed: 0|1`. The Phase 9 widget already renders `R▓` when set.
- Extension is purely a TOOL_SLOT_MAP addition + a new `slot === 'reaction'` branch in the createChatMessage hook body that sets `current.reactionsUsed = 1`.
- The `updateCombat` reset path already zeroes `reactionsUsed` on turn/round advance — no changes needed.
- 2024 PHB note: a player gets ONE Reaction per round (not per turn). dnd5e 5.3.3 tracks reactions on a per-actor basis through round transitions — the existing reset-on-turn-advance is technically over-aggressive (resets on every turn instead of every round), but this matches the existing tracker's behavior for Action/Bonus and is consistent. Refinement to "reset only on round advance" is deferred to a future milestone (not in 13-CONTEXT scope).

**Dispatch envelope shape (per Plan 13-01 schemas):**
- Shield Y-tap → `{toolId: 'cast-shield', args: {actor_id, slot_level: 1}}` (actor_id is the PLAYER actor; the panel reads it from `evf.session.playerActorId` exposed by the boot engine — see existing patterns in ActionOptionsModal which already threads currentActorId).
- Counterspell Y-tap → `{toolId: 'cast-counterspell', args: {actor_id, slot_level: 3, target_caster_id}}` where `target_caster_id` is the `sourceName`-correlated actor (resolved via existing `game.actors` lookup or — for the MVP — left as a TODO that surfaces via `<unknown>` sentinel if not resolvable; in V2 we accept this approximation).
- Opportunity Attack Y-tap → `{toolId: 'opportunity-attack', args: {actor_id, item_id, target_id}}`. `item_id` defaults to the player's primary melee weapon ID (look up via the existing CharacterSnapshot inventory — first item with type='weapon' AND (system?.range?.units === 'ft' OR similar melee detector). For MVP take the FIRST type='weapon' as the deterministic default; ambiguity resolution is post-milestone.
- IF actor_id is unavailable (no session.playerActorId yet wired) the prompt panel emits NO envelope and shows an error toast `error.reaction.no_actor` instead. Defensive behavior — must not crash.

**Dispatcher architecture:**
- `attachReactionPromptHandler(ws, layerManager, bridge, gestureBus, locale, getPlayerActorId, getPlayerWeaponId)` returns an unsubscribe closure.
- Module-scoped state: `let pendingTimerId: ReturnType<typeof setTimeout> | null = null; let mountedPanel: ReactionPromptPanel | null = null;`
- On envelope arrival: clearTimeout(pendingTimerId); if mountedPanel != null → drop; else schedule mount via setTimeout 500ms.
- On mount: panel constructed with payload + auto-timeout setTimeout(5000) that calls layerManager.bundle to destroy itself.
- On panel close (any path): clear both timers + null mountedPanel.
