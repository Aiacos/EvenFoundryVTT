<threat_model>
T-03-01 [HIGH]: **Gap-injection attack on WS resume** — a malicious or compromised client could intentionally send `client_resume {last_seq: K}` with a K chosen to land between buffer eviction boundaries, hoping to coax the bridge into emitting a partial replay that hides a real game-state mutation (e.g. a HP drop). Mitigation: `ReplayBuffer.hasGap(sessionId, fromSeq)` must return `true` if the buffered entries with seq > fromSeq are not strictly consecutive; on any `hasGap === true` the bridge MUST send `resume_full_snapshot {reason: 'buffer_gap'}` and MUST NOT emit the gapped envelopes. The client then re-fetches via REST (`GET /v1/character/:actorId`, `GET /v1/combat/current`, `GET /v1/scene/viewport`) where the snapshot is authoritative. No new full-state-dump WS message is invented (ADR-0002).
T-03-02 [HIGH]: **Stale-session memory exhaustion via abandoned connections** — if a WS socket closes without the bridge wiring `socket.on('close')` cleanup, both `SessionStore` and `ReplayBuffer` retain per-session state indefinitely. Mitigation: every accepted handshake MUST register a `socket.on('close')` listener that calls `deltaEmitter.unregisterSession(sessionId)` + `sessionStore.deleteSession(sessionId)` + `replayBuffer.clearSession(sessionId)`. Verified by unit test that closes a mock socket and asserts the three Maps shrank by one.
T-03-03 [MEDIUM]: **Malformed post-handshake messages crash the WS handler** — once the handshake completes, the `message` listener receives raw client input. JSON.parse can throw; Zod safeParse on non-object input returns `success:false`. Mitigation: `handleResume` wraps `JSON.parse` in try/catch and treats any non-`client_resume` message as a no-op (silently ignore). NEVER `throw` or `socket.close()` from this listener — Phase 4a will introduce other client message types (e.g. heartbeats) and we must not regress.
T-03-04 [LOW]: **Replay buffer leak on session-not-yet-recreated reconnect** — current `handleHandshake` reconnect branch checks `sessionStore.getSession(client.session_id)`; if the SessionStore entry was cleared by `socket.on('close')` but the replay buffer was NOT cleared, a reconnect would see `replay_seq = 0` while stale envelopes still occupy memory. Mitigation: `socket.on('close')` clears BOTH stores symmetrically (T-03-02 mitigation covers this).
</threat_model>
